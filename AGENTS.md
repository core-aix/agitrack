# aGiTrack Requirements

aGiTrack stands for agent + git tracking. It is a Python library and interactive CLI that combines coding-agent command line tools with automatic Git commits so agentic code changes are easier to trace.

## Goals

- Provide a common interactive interface for coding-agent backends and Git commit automation.
- Support OpenCode and Claude (Claude Code) as interchangeable backends; every aGiTrack feature works the same regardless of the selected backend.
- Keep the user experience similar to the selected backend CLI, while adding aGiTrack commands for agent switching, user commits, staging, status, and configuration.
- Make agentic code changes traceable by automatically committing code changes made after agent prompts.
- Support running aGiTrack from any folder against a target working repository.

## Documentation

- `docs/user-flow.md` holds the complete, graph-rendered map of aGiTrack's interactive user flow (Mermaid flowcharts: startup gating, base↔worktree commit/stage/sync, the agent turn and integration, copy-back, the Ctrl-G menu, self-update, exit). It is linked prominently from the README. **It must be kept in sync with the actual flow:** whenever the user flow changes — a new or reworded prompt, a changed option label, a new decision based on file/commit/session status, a new Ctrl-G command, or a changed exit/update path — update the matching diagram in the same change.

## Commit Types

- Agent commits use the `<aGiTrack>` tag.
- User-triggered commits use the user-provided subject without an aGiTrack subject tag.
- Before an agent acts, if user changes already exist, aGiTrack creates a separate user commit first.
- A commit is created only when code has changed and staged changes exist.
- **A slash command that does real file work must be committed and merged like any other turn.** Some commands are pure directives (`/model`, `/clear`, `/compact`) and produce no changes, but others — most importantly `/init`, which writes `CLAUDE.md`/`AGENTS.md` — drive the agent to edit files. Those file changes must be auto-committed and merged into the base just like a typed prompt. The trap is backend-specific transcript shape: **Claude** records a `/init` as a `<command-name>` artifact row PLUS a separate `isMeta` user row holding the command's expanded instructions, and the transcript parser drops both as non-prompts — so without special handling NO turn opens and the file-creating assistant work is silently dropped (no commit, no merge). `transcripts/claude.py` therefore detects a `<command-name>` invocation (`_slash_command_name`) and lets the following expansion row (`_command_expansion_text`) open a turn labelled with the command (e.g. `/init`); commands with no expansion still open no turn. **OpenCode** already opens a turn for any `role == "user"` message, so a completed `/init` is captured natively — but a turn only commits when it has a `final_response`, which OpenCode fills only once the run reaches its `final_answer`/`finish == "stop"` message, so an interrupted command run legitimately has none. Verified on both backends by running a real `/init` end to end (Claude writes `CLAUDE.md`, OpenCode writes `AGENTS.md`) and confirming the parser yields a committable turn. Keep this distinct from the trace-exclusion rule below: pure directive commands stay out of the trace; work-doing commands become real, committed turns.

## Manual-Commit Mode (`--manual-commits` / `-m`, config `manual_commits`)

- An opt-in, strictly additive mode (default off). It **always runs without a worktree**: enabling it forces `use_worktrees=False` in `cli.py` (`--manual-commits`/`-m` implies `--no-worktree`), so it never runs in the worktree model. Everything below runs only when `ProxyRunner._manual_commits` is true; the default worktree and existing `--no-worktree` paths are untouched.
- **Goal:** natural, user-triggered commits. The agent edits the current branch directly, but aGiTrack does **not** commit each turn onto the branch. Each turn is recorded as a hidden **latent commit** on `refs/agitrack/manual/<session_id>` (chained, first-parent = the branch HEAD / previous latent tip) while **HEAD never moves**. The working tree already holds the agent's edits, so the latent commit is a durable *record* of the turn (its snapshot tree + the usual `<aGiTrack>` subject + `# Interaction Trace` + `# aGiTrack Metadata`), used for the dashboard's pending view and for folding at commit time.
- **Latent sink.** `CommitEngine.commit_turns` takes injected `manual_gate_fn` / `manual_record_fn` (from `ProxyRunner._manual_gate` / `_manual_record`). In manual mode it skips `_ensure_turn_branch`, the `add_tracked`/`stage_untracked` staging, and the `has_staged_changes` gate — it never touches the user's index or HEAD. The gate is "did the working tree change since the latent tip" (compare `snapshot_worktree_tree()` to the tip's tree); the record does `commit_tree(snapshot, parents=[tip or HEAD])` + `update_ref`. Snapshotting uses a throwaway `GIT_INDEX_FILE` (`GitRepo.snapshot_worktree_tree`), so the real index/staging is left pristine, and drops the scaffolding dirs (`.agitrack/`/`.claude/`/`.opencode/`).
- **One commit at commit time.** When the user commits — via `Ctrl-G → git-commit` (renamed from `git-user-commit`) or an external `git commit` — a managed **`prepare-commit-msg`** hook appends the pre-rendered trailer to the message, and a **`post-commit`** hook resets the latent ref to the new HEAD, clears the trailer, and touches a signal file. So the single commit carries the user's changes **and** every pending turn's trace/metadata, with no separate cover commit and no `<aGiTrack>` commits polluting the branch. Hooks live in `agitrack/git/hooks.py` (`install_manual_commit_hooks`, same marker/chain/skip-on-`core.hooksPath` pattern as the base-commit guard) and are installed/removed in `_setup_manual_commit_mode` / `_teardown_manual_commit_mode`.
- **The trailer** (`build_manual_squash_trailer`, `commits/message.py`) is a *squashed* message: a leading `commit_type: user` block (so even a commit with no pending turns is attributed to the session) followed by each pending latent commit's full body. Because that yields several `# aGiTrack Metadata` blocks, the dashboard's existing squash parser (`metrics/collect._parse_constituents`) reads one constituent per turn, sums the tokens, and classifies the commit as agent-tracked — **no new `commit_type` was needed**. The runner re-renders the trailer file (`.agitrack/manual-pending-trailer`) and the ref-name file (`.agitrack/manual-ref`) from the durable ref every turn (`_render_manual_trailer`), so it survives restarts; the `git-commit` menu handler also folds the trailer inline so it's tracked whether or not the hook is installed.
- **Fallback (no hook).** When `core.hooksPath` is set the hooks can't be installed; `_service_manual_commit_mode` then polls HEAD (`_reconcile_manual_external_commit`) and, on a detected external commit with pending turns, adds a `cover_commit(..., tree=<new HEAD tree>)` carrying the trailer (its tree equals HEAD's, so it introduces no diff) and resets the ref.
- **Dashboard.** `collect_manual_pending` enumerates `refs/agitrack/manual/*` for commits HEAD doesn't contain and marks them `pending`; `build_dashboard` appends them when showing HEAD, and `web.py` renders a dashed `pending` badge. After the user commits, the ref resets to HEAD so nothing is double-counted (and squash-constituent dedup covers any transient overlap).
- **Recovery.** `_setup_manual_commit_mode` resets a stale latent ref whose tip is already contained in HEAD (an interrupted commit), so previously-folded turns are never re-folded.
- Tests: `tests/test_manual_commits.py` (primitives, trailer/squash parsing, hooks fold/idempotency/chain, CommitEngine sink, pending collection, config toggle).
- The manual latent machinery is factored into `agitrack.commits.ManualCommitTracker` (`commits/manual.py`) — `ref/gate/record/render_trailer/reset_stale_ref/service/reconcile_external_commit/setup/teardown` over a `(repo, base_repo, state)` bundle — so the headless **background** runner reuses the exact same logic. (`ProxyRunner` still carries its own `_manual_*` copy for now; both call the same GitRepo primitives + `build_manual_squash_trailer`.)

## Latent tracking = ALL no-worktree modes (fold hook, cover as backup)

- **Every no-worktree session records turns as hidden latent commits and folds them via the `prepare-commit-msg` hook** — not just manual mode. `ProxyRunner._latent_tracking` (property) is `not self._use_worktrees`; `_noworktree_auto` is `not use_worktrees and not manual_commits`. All the latent machinery (`_setup/_teardown_manual_commit_mode`, `_manual_gate/_record`, `_render_manual_trailer`, `_service_manual_commit_mode`, `_reconcile_manual_external_commit`, the `commit_turns` `manual_gate_fn/record_fn` routing, `backend_commits=[]`) is gated on `_latent_tracking`, so it runs for no-worktree AUTO too. Worktree mode is untouched (turns commit per-branch + integrate). The manual-specific **UI** (menu `git-commit` behavior, exit "commit" reminders, the pre-agent no-check fast path, the "pending" dashboard) stays gated on `_manual_commits` only.
- **No-worktree AUTO fold.** `ProxyRunner._auto_fold_latent_pending` (called from `_service_manual_commit_mode`, throttled) folds the pending latent turns into a real commit **itself** once a turn is done and the tree still holds the agent's uncommitted work — the branch advances per turn like normal auto mode, but the trace/metadata ride the SAME commit as the code (no separate cover). **The message is a CLEAN agent commit** built by `commits.build_auto_fold_message` — one pending turn ⇒ its own single-agent-commit body verbatim (subject = the prompt, or the LLM summary once `pending_bodies` folds it in; one `commit_type: agent` block), several ⇒ a squash led by the newest turn's subject. It must NOT use `build_manual_squash_trailer` here (that "squash into a user commit" format leads with a generic `<aGiTrack> commit agent turns` subject and a spurious `commit_type: user` block — wrong for aGiTrack's own agent commit). The background daemon uses the SAME helper and additionally **waits (bounded by `summary_wait_seconds`) for the summary note before folding** (`BackgroundRunner._fold_summary_ready`), since the daemon never amends HEAD, so the summary must be in the message before the commit, not amended after. When the AGENT instead makes its **own** commit, the installed `prepare-commit-msg` hook folds the pending tracking straight into that commit; a clean tree then makes `_auto_fold_latent_pending` a no-op. The old no-worktree cover pass (`_integrate_agent_made_commits_if_idle`'s `not _use_worktrees` branch) is skipped under `_latent_tracking` — `_reconcile_manual_external_commit` is the cover BACKUP (only when the hook can't run, custom `core.hooksPath`). This realizes "auto mode uses the prepare-commit-msg hook for the agent's own commits, cover as backup."
- Tests: `tests/test_manual_commits.py::test_noworktree_auto_*` (latent→fold, agent-self-commit folds via hook with no cover, reconcile cover backup when hook unavailable).

## Background Mode (`--background` / `-b`, config `background`)

- An opt-in, strictly additive **headless** mode (default off). It **always runs without a worktree** (`cli.py` forces `use_worktrees=False`; `--background` implies `--no-worktree`) and uses **auto** commits by default (like the TUI) — `-m`/`--manual-commits` (or `manual_commits: true`) opts into manual; `--auto-commit` forces auto over a configured manual. Enabled by `-b`/`--background` (const `"run"`) or the `background` config key (settable in both global and per-repo config); resolved in `cli.py` and dispatched to `agitrack.proxy.background.BackgroundRunner` **before** the TUI/json branches.
- **Not a proxy.** Background mode does NOT spawn or drive the agent. The user runs the agent from any UI; the runner polls the newest backend session transcript in the repo dir (`backend.latest_session_id`, so it follows an in-backend session switch) and records completed turns. This is the interactive-UI-agnostic tracker of issue #143.
- **Reuses the proxy pipeline.** `BackgroundRunner._process_once` builds a bare `Session` (`worktree=None`) and calls `CommitEngine.start_parse` + `finish_parse_if_ready` exactly as the proxy does — so turn detection, the per-conversation watermark, and token accounting are **identical**. The commit sink records turns latently via `ManualCommitTracker` (both modes); auto mode's `_auto_fold_pending` mirrors the proxy's `_auto_fold_latent_pending`. Summaries are written as git notes (`agitrack/commit-summary`) so the fold picks them up (never amends HEAD — the user may be committing concurrently).
- **Runs as a DETACHED daemon (like `agitrack -d`), not foreground.** `agitrack -b` is a *launcher*: `cli.py` splits on the internal `--background-serve` flag — without it (the launcher) it does the interactive privacy ack, then `background.start_background_daemon` re-execs `agitrack --background --background-serve --skip-privacy-ack …` detached via `proc.detach_kwargs()` (stdout/stderr → `<repo>/.agitrack/background.log`) and returns to the shell; the detached child (with `--background-serve`) runs `BackgroundRunner(...).run()` holding the lock. The launcher releases the lock before spawning so the child can take it. Commit mode is forwarded explicitly (`--manual-commits`/`--auto-commit`) so the child's own config can't flip it. **No owner-pid watchdog** (deliberately unlike the dashboard daemon): a tracker must outlive the launching terminal — stop it with `agitrack -b stop`. `start_background_daemon` reuses a running daemon (no duplicate) and waits on the handshake (`wait_for_handshake`, correlated on the child pid).
- **Single instance + stop/status.** Background holds the shared repo lock (`RepoLock` at `<repo>/.agitrack/lock`) for its whole run, so only one aGiTrack (TUI *or* background) runs per repo — a second start is refused (`already_running_message`). It also writes `<repo>/.agitrack/background.json` (its pid) so `agitrack -b stop` / `-b status` (handled early in `cli.py`, `stop_background`/`background_status`) target ONLY a background tracker (never a foreground TUI). Stop sends SIGTERM; teardown records any final turn (folding it in auto mode), removes the manual fold hooks (but NOT the persistent auto-track hook — see below), and deletes the handshake.
- **Repo scoping is mandatory.** A background daemon must track ONLY this repository's backend session, never one driven in another repo. This is inherent in `latest_session_id(self.repo.repo)`: Claude reads only `~/.claude/projects/<cwd-encoded>/` (the exact repo dir), OpenCode filters `session.directory == repo`. Keep that scoping — never widen discovery to "globally newest".
- **Event log (`--log-file` / config `log_file`).** Both the daemon and the proxy append notable events (`daemon-start`, `ai-change-detected`, `commit`, `update-available`) to a user-chosen file via `agitrack.events.EventLog` (best-effort, open+append+close per line, never raises). `resolve_log_path` resolves a relative path against the repo root. `EventLog(None)` is a no-op (used as the test default so `for_testing` runners have `.events`). Must work in every mode.
- Tests: `tests/test_background.py` (latent record + frozen HEAD, fold-on-user-commit, auto self-fold, skip-when-agent-committed, follows latest session and counts once, stop/status handshake, run() writes+removes handshake, detached-daemon launcher spawn/reuse/failure, event-log emission, update-marker write/clear, precommit-sync fold/no-op/defer/autostart, auto-pipeline end-to-end **regression guard**, direct `ManualCommitTracker`); `tests/test_events.py`, `tests/test_update_marker.py`; gating in `tests/test_cli.py`.

## Persistent auto-track pre-commit hook + auto-start (remind / auto-start on commit)

- **Problem:** a background tracker is easy to forget (e.g. after a reboot), so AI work committed while it isn't running would go untracked. **Solution:** a PERSISTENT `pre-commit` hook (`git/hooks.py`, `install_autotrack_precommit_hook`, marker `# AGITRACK-AUTOTRACK-PRECOMMIT`) that outlives aGiTrack. Installed in every no-worktree/background run (`ProxyRunner._setup_manual_commit_mode` and `BackgroundRunner._install_autotrack_hook`) and — unlike the manual fold hooks — **NOT removed on teardown**. It bakes in `sys.executable` + the repo root so it needs nothing on PATH, is a no-op inside a linked worktree, chains a pre-existing hook (and coexists with the worktree base-commit guard via the same `.agitrack-orig` backup — they're mode-exclusive), and always `exit 0` (never fails a commit).
- **`agitrack --precommit-sync`** (internal, `background.precommit_sync`): best-effort, always returns 0. If the repo lock is held (a live tracker) → do nothing (that tracker folds this commit). Else, when the AI made changes since the last commit (non-zero pending turns), it records the pending turns latently and renders the fold trailer + installs the manual fold hooks, **so the triggering commit itself folds in the trace/metadata** (`prepare-commit-msg` runs after `pre-commit`, so installing it during `pre-commit` still folds THIS commit). The triggering commit stays the author's own **manual** commit (they already wrote a message). A purely human commit (no pending turns) is left untouched (no footprint). Then, unless `autotrack_hook == "off"`, it **auto-starts** the detached daemon for the turns that follow (fire-and-forget, after releasing the lock) in the **same commit mode as the last run** (`read_background_mode`, persisted by `write_background_mode` at daemon start), and prints an explicit "started automatically in {auto/manual}-commit mode … stop with `agitrack -b stop`; disable with `agitrack --remove-hooks`" message. It also prints any available-update reminder on every commit.
- **No confusing "reminder-only" middle state.** The hook == auto-start: `autotrack_hook` (`"auto"` default / `"off"`) is the single repo-scoped control (the old `background_autostart` bool was removed/consolidated). `--remove-hooks` sets it to `"off"` so a later run won't reinstall.
- **Opt out from the CLI: `agitrack --remove-hooks`** (`git/hooks.remove_all_installed_hooks`) removes EVERY aGiTrack-installed hook — the persistent auto-track `pre-commit`, the worktree base-commit guard, and the manual-commit `prepare-commit-msg`/`post-commit` fold hooks — restoring any chained project hooks. `pre-commit` is checked twice (auto-track then guard) so a guard restored by the first removal is itself removed by the second.
- **`agitrack -b` asks about the persistent hook, once per repo.** Because the hook OUTLIVES the tracker, the `-b` launcher (interactive TTY only) explains it and asks a **binary** enable/off (`cli._maybe_prompt_background_hook`, keyed on `config.source("autotrack_hook") == "repo"`; default = enable). The choice is the repo-scoped `autotrack_hook` (`"auto"`/`"off"`, config + settings menu). Both installers (`BackgroundRunner._install_autotrack_hook`, `ProxyRunner._setup_manual_commit_mode`) honor it: `off` REMOVES any existing auto-track hook instead of installing. Never blocks automation (no-TTY/scripted ⇒ default enable, silently). The message tells the user how to cancel it (`agitrack --remove-hooks`).
- **`agitrack --status` / `-s`** (`background.repo_status`) reports whether aGiTrack runs for the repo and the mode: background (from `background.json`, always no-worktree) vs interactive (from a `session.json` handshake the proxy writes at `run()` and clears on teardown, since the shared repo lock carries only a pid), auto vs manual commit, worktree vs no-worktree — or not running. Read-only; dispatched early in `cli.py`.
- **Global `summarization_enabled: false` must not be shadowed.** `BackgroundRunner._summarization_enabled` reads the GLOBAL config (with repo overlay) FIRST — matching `ProxyRunner._summarization_enabled` — because `AgitrackState` defaults `summarization_enabled: True`, and that per-repo default would otherwise override a global/repo `false` (so the toggle never took effect in background mode). Fall back to state only with no global config (tests).
- **Auto-fold doesn't hang when the summarizer fails.** `BackgroundRunner._fold_summary_ready` waits for the summary note, but bails early (folds now, raw-prompt subject) once the tip's summary worker thread has FINISHED without producing a note (`_summary_threads` keyed by latent sha) — otherwise a failing summarizer would make every fold wait out the full `summary_wait_seconds`.
- **Never leak internal state into a commit.** The daemon and `precommit_sync` write `.agitrack/` files (trailer, ref) even in an otherwise-idle repo, BEFORE any `AgitrackState.save()` — so they call `state.ensure_local_ignore()` first (idempotent info/exclude setup), or a `git add -A` would stage aGiTrack's own state. Keep that call before writing any `.agitrack/` file.
- Tests: `tests/test_manual_commits.py` (autotrack hook install/chain/remove, worktree no-op), `tests/test_background.py::test_precommit_sync_*`, `tests/test_cli.py::test_autostart_optin_*`.

## Background self-update reminders (never auto-install)

- Installing a newer aGiTrack can't be fully automated (pip/pipx/brew/MSI), so a background daemon **never** self-updates — it only RECORDS an available update and REMINDS. `BackgroundRunner._maybe_check_update` runs a throttled (`timings.update_check_seconds`) `Updater().check()` on a worker thread; on `available` it writes the shared marker `agitrack/update/marker.py` (`<repo>/.agitrack/update-available.json`) + emits an `update-available` event, and clears the marker when up to date. The interactive proxy feeds the same marker (`_consume_update_check_result`). Surfaces that read it: `agitrack -b status`, the pre-commit hook (commit-time reminder), and the dashboard banner (`metrics/web._update_banner_html` via the `__UPDATE_BANNER__` placeholder). Gated by `check_for_updates`.
- Tests: `tests/test_update_marker.py`, `tests/test_background.py::test_daemon_update_check_*` / `test_background_status_shows_available_update` / `test_precommit_sync_reminds_about_update_on_every_commit`.

## Commit Messages

- Agent commit subjects start with `<aGiTrack>` followed immediately by the latest user query for readability.
- User commit subjects use the user-provided commit message.
- User commit messages are required; blank user commit messages are rejected.
- Agent commit bodies include the full interaction trace since the last code-changing commit.
- The interaction trace includes full user prompts and final agent responses.
- **Queued follow-up messages (sent WHILE the agent is working) must be captured.** Claude Code records a message the user queues mid-turn as a `type:"attachment"` row (`attachment.type == "queued_command"`, `commandMode == "prompt"`, `origin.kind == "human"`) — **NOT** a `type:"user"` row — so the normal user-prompt path never sees it and the follow-up silently vanishes from the commit trace (a real bug: several of the user's instructions were dropped). `transcripts/claude._queued_human_prompt` recognizes these and `parse_rows` appends the queued text to the turn in flight (Claude threads it into the same response, so it belongs to that turn, not a new one). Excludes slash directives / non-human origins. Verified against a real transcript. Composes with the proxy's submit-time capture: the `_same_prompt` word-overlap dedup means a prompt also recorded live doesn't double up.
- Do not include thinking tokens or intermediate responses.
- Bare slash commands (`/compact`, `/comp`, `/model`, `/clear`, …) are backend/TUI directives, not prompts, and must be kept out of the interaction trace. The transcript parser already drops them; the proxy's own submit-time prompt capture (`CommitEngine.record_user_prompt`) must apply the same exclusion (`_is_slash_command`), or a typed `/compact` leaks into the commit as a stray `## User` / `/comp` entry — which is also redundant with the compaction lead-in note the trace already carries.
- Commit subjects and bodies must not contain terminal escape sequences or control characters; strip arrow-key/escape residue both where the prompt is captured and when building the message.
- Commit bodies include metadata such as backend, session name, backend session ID, aGiTrack session ID, model, commit type, and timestamps.
- Agent commit metadata includes the current context token count and token usage accumulated since the last code-changing commit.
- Token counts must be exact for every category (input, output, cache read/write, reasoning, and their sub-agent equivalents): count each backend message's reported usage exactly once — not more, not less. Claude splits one assistant response across several transcript rows that repeat the same usage (de-duplicate by message id), and sub-agents are recorded separately (own transcript files for Claude, child sessions for OpenCode) and must be fully included. The token counts are a headline metric, so changes to transcript parsing must be verified against real backend data, not only mocks.
- **Exactly-once holds across sub-agents and session switches.** A Claude sub-agent whose `agent-*.meta.json` is missing/unreadable has no parent tool id (keyed `None`); it must NOT be attributed to "the latest turn" on every parse (it would re-attach to each new turn and be committed again). Instead attribute it to the turn active at its file mtime (`_subagent_unmatched_mtime` → `parse_rows(unmatched_subagent_time=…)` → `_turn_index_at_time`): a STABLE home turn the commit watermark can trim after it is counted once. Likewise the commit watermark is **per conversation** (`AgitrackState.backend_message_id_for` / `set_backend_message_id`): the conversation currently tracked (`backend_session_id`) uses the single `last_backend_message_id` (unchanged behavior, and every existing reset still governs it), while a DIFFERENT conversation the user switched to inside the backend reads its own remembered mark — so switching between conversations never replays/double-counts a conversation's committed turns. No-worktree mode now also FOLLOWS an in-backend switch: the parse worker prefers snapshot-based discovery (`_discover_spawned_session`, which returns None without a reliable pre-spawn snapshot so it can never grab a pre-existing unrelated session) over the pinned id.
- A turn's `input` is *fresh* input = uncached input **+ cache-creation (`cache_write`) tokens**, because cache-creation tokens are input the model processed once on the way into the cache. This is deliberately different from the provider's billing model (Anthropic prices cache writes, cache reads, and uncached input as three separate line items at different rates); aGiTrack folds cache-creation into `input` so the number answers "how much fresh context did this turn process?" rather than mirroring the price sheet. Keep the raw breakdown recoverable: still record `cache_write` (newly cached) and `cache_read` (replayed from cache, never folded into `input` — it was already counted when first written) on their own lines. The summarizer's cost obeys the same rule (`summary_tokens_input` includes its cache-creation; `summary_tokens_cache_read` is separate). The same fact is documented for users in the README token-accounting section — keep the two in sync.
- The summarizer must read **only** its instruction prompt plus the interaction trace (the same `## User` / `## Agent` text the commit carries) and nothing else. Run the summarizer backend in `bare` mode — a plain text completion with no tools, no project/user memory or MCP servers, and the agent's default system prompt replaced with a minimal one — so the summary isn't charged thousands of input tokens of system prompt and tool schemas it never uses (a real Claude call dropped from ~18,000 input tokens to a few hundred). `bare` is plumbed through `AgentBackend.run(..., bare=...)`; the main coding agent (and shell mode) must NOT use it, since it needs its tools. Keep this scoped to the summarizer.
- The summarizer is a mechanical text-reduction task, so its `bare` run also drops model thinking/reasoning to the floor (it gains nothing from it and it only adds latency and tokens). Each backend uses its own lever: Claude sets `MAX_THINKING_TOKENS=0` in the child env (a true off on the Anthropic API; a caller-set value still wins); OpenCode passes `--variant minimal` (the lowest reasoning effort its CLI exposes — it has no full "off"). Both are scoped to the `bare` path only and applied best-effort (a backend/provider that ignores the lever just runs as before); coding sessions keep their default thinking.
- Record reasoning/thinking token counts in commit metadata only when the backend session record reports them; otherwise omit the reasoning line. Do not add explanatory token notes to the metadata.
- Record the conversation's reasoning effort / thinking level as a `reasoning_effort:` metadata line, **only when the transcript reveals it** — neither backend records a numeric budget, so this is a coarse, best-effort signal, never asserted as "off". It lives per-turn on `SessionTurn.reasoning_effort`: Claude sets `"on"` when a turn carries a `thinking` content block (the sole transcript signal that extended thinking was active); OpenCode prefers a named effort/variant from the assistant `info` when the export carries one (`_REASONING_EFFORT_KEYS`) and otherwise falls back to `"on"` when the turn spent reasoning tokens. The commit takes the most recent turn that recorded a level (a later `None` does not erase it), so the metadata reflects the level in effect at the end of the span; the line is omitted entirely when no turn revealed one. This is the *coding* conversation's level — distinct from the summarizer, which is forced to its floor on the `bare` path.
- The interaction trace records only each turn's **final** agent reply by default. Including every user-facing message the agent sent during the turn (each as its own `## Agent` block) is opt-in two ways: the per-repo `full_agent_messages` config option (`.agitrack/config.json`, default off, mirroring `trace_turn_limit`), or the `--full-agent-messages` CLI flag which forces it on for that run only. The flag is a per-run override carried as `ProxyRunner._full_agent_messages` and passed to `CommitEngine(full_agent_messages=...)`: `None` defers to the per-repo config, `True` forces it on without persisting. The full list lives on `SessionTurn.agent_messages` (always populated by the parsers, cheap); the engine chooses all-vs-final via `_agent_messages_for`. Only conversational replies are ever included — never tool calls, tool results, reasoning, or file edits — and the trace falls back to `final_response` when a backend didn't recover the full list.
- Record a `conversation_anchor:` metadata line pointing at the backend message id of the last turn the commit covers, **only when the transcript exposes one**. Combined with `backend:` / `backend_session_id:` it pinpoints the exact place in the (locally kept or shared) transcript these changes came from, so a session can be backtracked to that moment. The commit takes the most recent covered turn that carries an `assistant_message_id` (the same id used as the parse watermark); the line is omitted when the backend exposes none (e.g. the shell path). Writer-only — the dashboard does not consume it.
- Track events that reshape the token accounting in both the interaction trace (as a lead-in note) and the metadata: context compactions (Claude's `isCompactSummary` rows, OpenCode's `summary`/`compaction` assistant messages — `context_compactions: N`, attributed to the turn whose context the compaction shrank) and a session being forked or copied from another conversation (`forked_from:` / `copied_from:` with the source and, for a copy, the contributor). A compaction resets the context and a fork/copy inherits another conversation's context and tokens, so these must be recorded to keep the token counts interpretable. The fork/copy origin is a one-shot, surfaced by the new session's first agent commit and then cleared.
- Proxy mode must baseline continued OpenCode sessions on startup so old turns do not inflate token usage for the next commit.
- When the user cancels a turn (Esc) and it produced no committable response, the agent's partial edits would normally sit uncommitted. In the live interactive path the commit engine calls back into the runner (`on_cancelled_fn` → `_handle_cancelled_turn`) which, if the worktree has changes, offers keep / commit-now / discard (discard is confirmed and runs `GitRepo.discard_all_changes`). "Commit" reuses the normal turn-commit path; on commit or discard the parse watermark advances past the cancelled turn. A "keep" answer is remembered in-memory (`_cancel_prompted`, keyed by the turn's backend message id) so it isn't re-prompted every parse cycle. Only the live path prompts — the sync/exit finalize paths pass `on_cancelled_fn=None` so no modal is raised during teardown.

## Staging Behavior

- When starting a coding agent, if the backend CLI supports appending to its system prompt (Claude's `--append-system-prompt`; OpenCode's TUI has no such flag), append a note that the session runs inside aGiTrack, which auto-commits each turn, so the agent must not create git commits itself unless the user explicitly asks. Apply this to coding sessions only (interactive proxy spawn and the shell-mode `run`), NOT to the summarizer's bare run. It is on by default but can be disabled per run with `--no-commit-guidance` (or `commit_guidance: false` in the global config), threaded as `commit_guidance` through the runners → `spawn_command`/`run`.
- The note is built by `agent_system_note(use_worktrees=...)` (`agitrack/backends/proxy_agents.py`). In the **default worktree model** it also tells the agent that **its current working directory IS the session worktree** (under `.agitrack/worktrees/`), that it should make its edits there and **not** edit the base repository checkout directly, and that aGiTrack handles **both the commit and the merge** of those worktree edits into the current branch. That clause is **omitted under `--no-worktree` and in shell mode** (the agent edits the current branch directly). The proxy spawn threads `use_worktrees=self._use_worktrees`; shell mode uses `use_worktrees=False`. `AGENT_SYSTEM_NOTE` remains as the worktree-model constant. **Word this carefully:** an earlier phrasing ("you do not need to … clean up anything under `.agitrack/`") was read by the agent as "don't write under `.agitrack/`" — so it edited the base repo checkout instead of its worktree cwd, and that work was never committed or merged. The note must make explicit that the worktree IS the cwd and edits belong there. The cwd itself is set correctly by the runner: `make_child_process(command, str(self.repo.repo))` passes the worktree as the child's working directory for both backends (on Windows via `CreateProcessW`'s `lpCurrentDirectory`), and OpenCode additionally receives the worktree as a positional CLI arg. Locked by `test_proxy.py::test_spawn_sets_backend_cwd_to_the_session_worktree` and `::test_opencode_spawn_command_embeds_the_worktree_path`.
- **Custom backend launch command (wrapper).** The backend executable that heads the spawn command (`claude` / `opencode`) can be replaced by a user-supplied launch command so the agent runs under another tool — e.g. `--backend-command "somewrapper claude"` produces `somewrapper claude …`, extending the verbatim-passthrough feature (#32) on the *front* of the command the way `backend_args` extends the *back*. It is threaded as `executable=` into `ProxyAgent.spawn_command` (which builds its head from it, defaulting to `[<binary>]`) and as `launch_command=` into the headless `ClaudeBackend`/`OpenCodeBackend` constructors, so it applies everywhere aGiTrack execs the agent: interactive proxy spawn, shell-mode `run`, and the summarizer. aGiTrack's own sandbox wrapper (`_confine_to_worktree`) still goes **on top** (outermost), and the backend's own flags + `backend_args` are appended **after** the wrapper head. Resolution: a per-run `--backend-command` (shell-split in cli.py via `_resolve_backend_command`, which fails fast on a malformed value) wins; otherwise `GlobalConfig.backend_command(backend)` — a string (any backend) or an object keyed by backend name (so a mid-session backend switch picks up the right wrapper). The runner/shell resolve it per spawn via `_launch_command()` because the active backend can change at runtime. A user-supplied launch command **bypasses** the backend install gate (`ProxyRunner._ensure_backend_available` returns early when `_launch_command()` is set) — a wrapper replaces the backend binary, so the default CLI need not be on PATH. Empty ⇒ launch the binary directly. Because aGiTrack tracks transcripts/sessions for the **selected** backend, `_confirm_backend_command_mismatch` (cli.py) warns and requires an explicit `y` when the command's tokens clearly name a *different* known backend than the selected one (basename match against `available_backends()`); an opaque wrapper that names no backend is accepted silently, and a non-interactive/scripted run proceeds with the warning rather than hanging on a prompt.
- In an isolated worktree session, offer to copy any files the agent left UNCOMMITTED in the worktree (declined untracked files, unstaged edits) or that are git-ignored into the base repo directory — they otherwise never reach where the user works (`_offer_copy_unstaged_to_base`). **Before the copy offer, `_offer_user_commit_for_worktree_edits` offers to COMMIT the user's own uncommitted worktree edits** (gated on `actions.has_pre_agent_user_changes()`) — those belong in git, not just copied — so when both a user edit and copy-able leftovers exist, BOTH prompts appear (commit first, then copy). It's a no-op in the normal turn case (the turn's auto-commit already committed tracked edits). This runs for the **active** session only (a background session is never interrupted mid-run); its files are caught instead when you **switch to it** (`_switch_active`, `context="switch"`) or on **aGiTrack exit just before the worktree is deleted** (`_remove_worktree_on_exit`, `context="exit"`, interactive exit only — skipped on a signal teardown where `screen is None`). The offer popup lists files **vertically and scrollably** (`SelectModal(detail=…)`, PgUp/PgDn). A file already accepted/left in place isn't re-offered until its content changes (`_copy_prompted` fingerprint); **declining mutes the whole current SET of paths** (`_copy_declined`) — aGiTrack won't re-ask while only those files keep changing, and a genuinely new path re-opens the whole set; the mute clears on session switch and aGiTrack restart, and the popup says so. The **exit** offer ignores the mute and warns that declining discards the files. Files that would overwrite existing base files get a single up-front choice — overwrite all / keep all / confirm each one individually. Anything not copied is reported as remaining in the worktree, with the worktree path. No-op without a worktree.
- The reverse direction (base → worktree) runs **before forwarding a prompt** (`_pre_agent_commit_if_needed`): the user's own edits live in the BASE repo working tree (the worktree is the agent's sandbox), so aGiTrack commits the uncommitted work from whichever tree holds it — this session's worktree (`_create_user_commit_popup()`, onto its turn branch) and/or the base repo (`_commit_base_user_edits_if_needed`, onto the base branch), both when both are dirty — then `_sync_idle_worktrees_to_base` merges/re-points the base commit into the worktree(s) so the agent starts from the user's edits, and only then is the prompt sent. Base detection (`_base_user_edits_pending`) counts any tracked edit **or any new untracked (added-but-unstaged) file** (`untracked_files()`, which excludes `.agitrack/`); the user is offered to stage it at the `[y/N]` prompt and, on yes, it's committed and synced. A "don't stage" answer only suppresses re-prompting for the same tree state via a fingerprint gate (`_base_edits_declined_status`, remembered until the base tree changes), so a file is always offer-able again and can never be permanently stranded. Explicit base commits (the `git-user-commit` command and the pre-prompt offer) pass `include_declined=True` to `_create_user_commit_popup` so the stage prompt re-offers every untracked file; the automatic worktree capture keeps `False` so the agent's own untracked decline still sticks.
- Worktree confinement (`agitrack/proxy/sandbox.py`, `_confine_to_worktree`) wraps the backend so its writes stay inside the session worktree (`sandbox-exec` on macOS, `bubblewrap` on Linux), keeping the base repo and sibling worktrees read-only. **It must never block the backend agent from updating itself.** Claude Code and OpenCode self-update in place by rewriting their own install/state/config/download dirs (`~/.claude`, `~/.local/share/claude`, `~/.opencode`, npm-global/Homebrew prefixes, …). `agent_writable_dirs()` enumerates those (realpath-resolved, derived from HOME/XDG + each CLI's resolved location on PATH) and both the macOS profile and the Linux bwrap command carve them out as writable — the macOS profile re-allows them after the base deny (later rule wins), and bwrap re-binds read-write any that fall **under** the read-only base (dirs outside base are already writable via `--dev-bind / /`; only existing ones are bound, since bwrap errors on a missing source). The sandbox confines the *base repo*, not the agent's own toolchain. Confinement is gated by the effective `sandbox` setting (config, or `--no-sandbox` for one run), resolved in cli.py and carried as `ProxyRunner._sandbox`. Users can also grant extra write carve-outs via `allowed_edit_paths` (config list) or `--allowed-edit-paths <p>[:<p>…]` (PATH-separated; CLI wins), carried as `ProxyRunner._allowed_edit_paths` and threaded through `sandbox.wrap_command(..., allowed_paths=…)` → `build_profile`/`build_bwrap_command` (re-allowed like the agent dirs; macOS covers non-existent paths, bwrap binds only existing under-base ones).
- Use `git add -u` by default for tracked modifications and deletions.
- When new untracked files are present, ask whether they should be staged.
- If the user declines staging untracked files, remember those files in repository-local state and do not ask about them again automatically.
- Inform the user when intentionally unstaged files exist.
- Provide an interactive CLI command to review and stage intentionally unstaged files.

## Repository-Local State

- Store state in `.agitrack/state.json` in the target repository.
- Ignore `.agitrack/` by default.
- State includes the aGiTrack session ID, selected backend, selected model, backend session ID, per-backend session IDs, declined untracked files, and pending interaction trace.
- Optional repository-local config lives in `.agitrack/config.json`; `trace_turn_limit` defaults to `5` and controls the maximum recent user turns included in an agent commit body.

## MVP Interface

- `agitrack` starts proxy mode in the current repository, launching the native OpenCode TUI through a pseudo-terminal and rendering it through an internal terminal screen with an aGiTrack status line.
- `agitrack --repo PATH` starts the interactive CLI for another repository.
- `agitrack --mode json` uses the structured JSON prompt-loop fallback.
- `agitrack --mode json --ui-bridge` runs a long-lived **bidirectional JSON-RPC bridge** over stdin/stdout (`agitrack/shell/bridge.py`: `BridgeServer` + `BridgeUI`) for programmatic GUI drivers. The driver sends `prompt`/`command`/`answer`/`exit` lines; aGiTrack streams `ready`/`response`/`commit`/`no_changes`/`notice`/`error`/`turn-complete`/`bye`, plus `ask` events (`kind`: select/multiselect/input/confirm) when it needs interactive input. Bridge mode is interactive even though stdin isn't a TTY: `AgitrackShell` injects `BridgeUI` into `AgitrackActions.ui`, so the untracked-file review and user-commit-message prompts (and the summarizer-model prompt) are asked of the driver instead of `input()`. `BridgeUI._ask` blocks on a thread-safe queue until the matching `answer` arrives. The terminal/`--prompt` paths are unchanged (`ui` defaults to `None`). `_emit`/notices route through the one lock-serialized bridge stdout channel. Covered in `tests/test_bridge.py`. NOTE: the VSCode extension in `editors/vscode/` does NOT use this bridge — it is a thin launcher that runs the real proxy-mode CLI in an integrated terminal (full parity). The bridge/`--json-events` remain as standalone programmatic interfaces. The TypeScript extension is not exercised by the Python gate (compile/package/F5 only).
- `agitrack --verbose` shows aGiTrack diagnostic messages; normal mode should avoid debug/status chatter.
- Plain text input is sent to the active agent backend.
- In proxy mode, all printable input is forwarded to the backend; aGiTrack controls are opened with `Ctrl-G` only. `:` is not an aGiTrack command trigger in proxy mode and is forwarded to the backend like any other character.
- Proxy mode command palette previews aGiTrack commands; Up/Down selects, Tab completes, and Enter runs the selected command.
- In JSON mode, aGiTrack commands use `:` instead of `/` so OpenCode-native slash controls are not intercepted.
- The interactive UI should show status information and contextual command hints for both `:` aGiTrack controls and `/` OpenCode-native controls.
- Intentionally unstaged-file notices should live in the status bar, not in the main transcript.
- Proxy mode renders the backend screen itself, so it must reproduce each cell's colors and attributes (bold/italic/underline/reverse) exactly as the backend emitted them.
- **Faint (SGR 2) must be reproduced.** pyte 0.8.2 has no slot for faint and silently drops `\x1b[2m`, so dim text would re-render at full intensity — most visibly Claude's grey autosuggestion ghost text, which then looks identical to what the user typed (#113). `_BackgroundColorEraseScreen` carries a `dim` flag on an extended Char (`_DimChar`): it overrides `default_char`/`reset` so every cell has the slot, tracks faint in `select_graphic_rendition` (2 sets it; 0/22 clear it; last wins), clears it on background-colour-erase, and `cell_sgr` re-emits `2`. **The `2` must be parsed positionally, exactly as pyte consumes SGR params — never matched anywhere in the list:** in a 24-bit colour like `38;2;r;g;b` (or `48;2;…`) the `2` is the colour selector, not faint, so `select_graphic_rendition` skips 38/48's parameters before testing for faint. Missing this dimmed all normal truecolor text (e.g. the user's own white input), not just Claude's suggestion. Any future pyte upgrade that natively supports faint should retire this shim.
- **Mouse: aGiTrack only owns the wheel; text selection is the terminal's.** For a backend that doesn't drive the mouse itself (Claude), aGiTrack enables host mouse reporting (`\x1b[?1000h`) purely so the **wheel** scrolls its rendered history (`_handle_mouse` → `_scroll`); PageUp/PageDown do too. It must **not** act on left-button press/drag/release — it used to drag-select-and-copy, but that path only ran in terminals that forward a plain drag to the app under mode 1000, where it both suppressed the terminal's native selection and popped an unwanted "Copied N char(s) to clipboard" message (#112). Terminals that select natively never forward the drag, so they were unaffected; dropping the behaviour removes only the harmful case. Mouse report bytes are still stripped from the input forwarded to the backend (`_intercept_scroll`). OpenCode drives its own mouse (`child_mouse`), so none of this applies to it.
- Proxy mode must re-emit colors in the same encoding/depth the backend used, chosen from the shared terminal color support (truecolor stays 24-bit; 256-color stays a palette index so the host terminal's own palette renders it; named ANSI stays named). Upconverting 256-color output to truecolor breaks terminals without truecolor support (e.g. Apple Terminal) and shifts colors on terminals with customized palettes.
- Proxy mode must answer the terminal capability queries the backend makes (foreground/background via OSC 10/11, palette via OSC 4, cursor position, device attributes) using the host terminal's real responses, so the backend detects the same theme it would in a native session. Without this the backend cannot match the host terminal's light/dark theme and colors render wrong.
- Proxy mode commands after `Ctrl-G` (bare names, no `:`), in this order: `sessions`, `agent-backend`, `summarizer`, `git-unstaged`, `git-user-commit`, `dashboard`, `update`, `exit`. Order matters because the palette selects by typed prefix.
- Only `sessions` starts with `s`, so pressing `s`+Enter jumps straight to the session picker. Git-specific commands are grouped under a `git-` prefix.
- `sessions` opens an interactive menu of the live concurrent sessions to switch between them, start a new one (in its own worktree; under `--no-worktree` it shares the base directory instead), or stop one. Each entry shows whether the session is `running` (a turn is in flight / it produced output recently) or `idle`. Typed forms: `sessions new`, `sessions <n>` (switch to the n-th).
- **`--no-worktree` allows multiple sessions too.** `_new_session` branches on `_use_worktrees`: with worktrees it opens/creates a worktree; without, the session runs on the base tree (`worktree=None`, `repo=self.base_repo`, `state=AgitrackState(base_repo.repo)`), committing to the checked-out branch. They share the directory with no isolation (the per-turn commit captures whatever is in the tree); `_warn_parallel_no_worktree_sessions` shows a one-time caveat. A blank no-worktree session resets the shared base state to a fresh conversation; resuming one routes through `_stage_backend_resume`, which (for a conversation first run in a worktree) retargets the transcript's recorded cwd to the base dir via the backend's `retarget_working_dir`, so it runs in the base directory rather than the gone worktree path.
- `agent-backend` selects the agent backend (`opencode` or `claude`); with no argument it shows a picker. Switching relaunches the backend TUI, restores that backend's previous session for the repo if known, and updates the saved global default.
- `git-status` shows Git status; `git-stage` reviews/stages untracked files; `git-unstaged` shows intentionally unstaged files; `git-user-commit` creates a user commit.
- `dashboard` serves the metrics dashboard (`agitrack/metrics` `build_server`) as a **separate background process**, not an in-process thread (#110) — so recomputing metrics from `git log` on every poll charges its CPU/IO/energy to that child, not the TUI. `agitrack/metrics/daemon.py` owns the lifecycle: the launcher spawns `agitrack --dashboard-serve --dashboard-owner-pid <pid>` (detached via `start_new_session`), the child binds its port and publishes a handshake file (`<repo>/.agitrack/dashboard.json`, atomic write), and the launcher reads it back (`wait_for_handshake`, correlated on the child pid) to learn the URL. The dashboard is **killed when its owner exits** by two mechanisms: the owner terminates it explicitly (the TUI's `_stop_dashboard()` in the `run()` finally; `agitrack -d stop`), and the child runs an owner-death watchdog (`_watch_owner`, `pid_alive` poll every `_OWNER_POLL_SECONDS`) that shuts the server down if the owner pid vanishes — covering a SIGKILL of the owner where no explicit stop runs. For the Ctrl-G dashboard the owner is the TUI process (`os.getpid()`); for `agitrack -d` it is the **launching shell** (`os.getppid()`), so it dies when that terminal closes. CLI verbs: bare `-d`/`-d html` start (or reuse) the background daemon and return to the prompt; `-d stop` / `-d status` manage it; `-d text` is still a one-shot foreground report. The browser is opened only when it would land on THIS machine: `metrics.server.browser_is_local()`/`open_dashboard_in_browser()` skip opening on a remote/SSH/Mosh/headless host (honoring `$BROWSER`), and `remote_browser_hint()` tells the user to use the forwarded URL. The launcher passes `_dashboard_email_logins()` (current user's git email → GitHub login) into the child via the `AGITRACK_DASHBOARD_EMAIL_LOGINS` env var → `build_server(..., email_logins=)` so the session's fresh, unpushed commits — which `gh` can't map until they're on the remote — still show the user's GitHub ID. The VSCode extension also exposes `agitrack -d` as an "Open Dashboard" command (VSCode forwards the port to the local browser).
- **Dashboard by-backend / by-model exclude non-informative labels.** `Dashboard.group_by` (`agitrack/metrics/collect.py`) omits a turn whose backend/model is not a real value rather than bucketing it under "unknown"/"<synthetic>". Two layers: (1) the metadata is normalized at parse — `_build_stat`/`_constituent` run `backend`/`model` through `_real_metadata_label`, which maps the writer's "unknown" placeholder (`commits/message.py` writes `model: {model or 'unknown'}`) and Claude's `<synthetic>` marker to `None`; (2) `group_by` then simply skips falsy labels. The root cause of `<synthetic>` is fixed at the source too: `<synthetic>` is the literal model Claude Code stamps on synthetic (non-LLM) assistant messages (compaction/interrupt markers), so the transcript parser (`transcripts/claude.py`, `SYNTHETIC_MODEL`) and the headless `_model()` selection (`backends/claude.py`) refuse to let it become the turn's model — the real model is kept. The parse-time normalization still scrubs already-committed history that recorded the placeholder before this fix.
- Closing the terminal (SIGHUP/SIGTERM — the VSCode/SSH/Mosh terminal closed, including a system restart) runs `_handle_exit_signal`, which now best-effort `_finalize_pending_work()` (suppressing render by nulling `self.screen`, fully guarded) before teardown, so a just-completed turn is committed/integrated rather than stranded. The VSCode extension opens sessions as editor-area terminals so VSCode's `confirmOnKill` prompts before the close.
- **A busy resumed session forks instead of crash-looping; other launch failures are surfaced (#114).** aGiTrack resumes the last claude session by default. When that session is still held by a running background agent, `claude` refuses to attach and exits, printing a message containing "running as a background agent"/"--fork-session". In proxy mode that error lands on the alt-screen, which teardown discards, so without handling the runner just crash-loops (`_relaunch_backend_or_exit` relaunches up to 3×) and exits with a silent flash. Fix (from PR #115): `_relaunch_backend_or_exit` first checks `_should_fork_after_busy_exit()` — claude backend, a recorded `backend_session_id`, that message in `last_child_output_sample` (`_FORK_HINT_RE`), and not already forked — and if so sets `_fork_next_spawn` and re-spawns **once** with `--fork-session` (claude's own remedy: branch off a copy, leaving the busy agent untouched). `_spawn` snapshots existing session ids when forking so the new id is discovered/adopted on the first parse, exactly like a backend that assigns its own id; `_forked_for_busy` guards against re-forking, so a fork that also dies falls through to the normal crash-loop guard. As a safety net for cases forking doesn't cover (other backends, other launch failures), the crash-loop bail builds `_backend_exit_notice` via `_format_backend_exit_notice()` (the backend's last readable output + how to get unstuck, e.g. `--new-session`), which `run()` prints on the restored host screen instead of leaving only a flash. `spawn_command` gains a `fork` parameter on all backends (claude appends `--fork-session`; opencode accepts and ignores it — no fork concept).
- **Forced-exit prompt (host terminal closed mid-work).** Because the TUI is already gone when SIGHUP/SIGTERM arrives, the user can't be asked anything *inside* the terminal, so `_handle_exit_signal` surfaces a blocking confirmation through the OS window server instead — but only when the close interrupted live work. It captures `_had_unfinalized_work()` (agent turn in flight, a background session running, or any tracked tree dirty) **before** finalizing (which commits that work away), then after teardown calls `_prompt_forced_exit()`. That delegates to `agitrack/proxy/host_prompt.py`: `can_show_dialog()` gates on macOS + `osascript` (SSH/headless/Linux stay silent and fast — no window server), and `confirm_forced_exit()` runs an `osascript display dialog` with a "Reopen aGiTrack"/"Quit aGiTrack" choice and `giving up after 25`, so a system restart is never blocked or hung (timeout/dismissal/any error ⇒ `QUIT`). "Reopen" sets `_reopen_after_exit`; once `run()`'s teardown has released the management lock, `_reopen_in_new_window()` opens a fresh Terminal window running `python -m agitrack <argv>` in the repo root (so the last session auto-resumes). Everything here is best-effort and guarded — a clean close, or any platform without a GUI, exits exactly as before.
- **GitHub sign-in from the VSCode extension.** aGiTrack uses `gh` for the dashboard's committer identities and session sharing, and its TUI warns when `gh` is unauthenticated and says to run `gh auth login` — but its full-screen TUI owns the terminal, so there is no shell prompt to run that in, and a GUI-launched VSCode's terminal child often can't read `gh`'s credentials at all (e.g. a token in the macOS Keychain), so the user gets stuck with no way to authenticate. The extension (`editors/vscode/`) must therefore provide a way to sign in: a `agitrack.ghLogin` command ("Sign in to GitHub (gh)") opens a plain terminal running `gh auth login` (run in the shell so `gh` resolves from the user's real PATH and writes credentials that environment can read), and after a session launches `maybePromptGithubSignIn` shows a one-click "Sign in to GitHub" notification **only** when it is actionable — `gh` installed but unauthenticated (strict `gh auth status` exit-code check, mirroring the CLI's `gh_status()`), the repo has a GitHub remote, the user hasn't permanently dismissed it, and it hasn't already prompted this activation. Pure decision logic lives in `editors/vscode/src/github.ts` (`shouldPromptGithubSignIn`, `hasGithubRemoteUrl`) with tests in `github.test.ts`; the terminal/notification plumbing is in `extension.ts`. Backend-agnostic (it concerns `gh`, not claude/opencode).
- **Startup gh-availability prompt (before the TUI).** Because aGiTrack's full-screen TUI takes over the terminal, the gh warning must be surfaced *before* it starts, while stdin is still an ordinary shell. `cli._check_gh_availability(repo)` runs right after the privacy acknowledgment (proxy mode only, not json/bridge): when `gh` is missing or unauthenticated (strict `gh_status()`) **and** the repo has a GitHub remote (`commit_url_base` non-empty — gh only matters there), it prints why gh matters and prompts the user to **log in now** (`'l'` → `gh auth login` runs inline in the still-cooked terminal), **continue without it** (Enter), or **quit** (`q`). It returns `(proceed, handled)`: `proceed=False` only on quit (caller aborts startup, releasing the lock); `handled=True` whenever the prompt was shown, which is threaded to `ProxyRunner(gh_prechecked=…)` so `run()` skips its background in-TUI gh notice (no double-nag). Never blocks automation: without an interactive TTY, or in scripted mode, it does nothing and returns `(True, False)` (the in-TUI transient notice still applies). Drains injected input first (like the privacy ack) so a stray newline can't auto-answer. This is the CLI half of the GitHub sign-in story; the VSCode-extension command/notification above is the other half.
- `exit` exits (with confirmation; finalizes pending commits first).
- **Startup menu-key conflict check + pre-TUI key test (before the TUI).** The aGiTrack menu key (default Ctrl-G) can be claimed by the host editor/terminal — notably VS Code binds Ctrl-G to "Go to Line" — so it never reaches aGiTrack; once the full-screen TUI starts there is no way to open the menu if its key doesn't work. `cli._verify_menu_key(config)` runs right after the gh check (interactive proxy launches only): `settings.detect_menu_key_conflict(key, env)` flags known host conflicts (conservative — currently VS Code + Ctrl-G via `host_is_vscode`), and when one is found (and not already acknowledged for this key) it lets the user **[t] test** the key, **[c] choose a different one** (with `suggest_menu_keys` offering rarely-grabbed letters), **[Enter] keep it**, or **[q] quit**. The **test is authoritative**: `_read_menu_key_press` puts the terminal in raw mode and waits for the exact bytes (`settings.menu_key_bytes_for`) the TUI matches against stdin — True if they arrive, False on timeout (the host swallowed the key), None if cancelled/no-tty; a key intercepted by VS Code simply never arrives, so the test fails exactly when the TUI would. A chosen key is persisted as `menu_key` (global) and the resolved key is stored as `menu_key_acknowledged` so the next launch stays quiet. Never blocks automation (no-TTY/scripted ⇒ returns True silently). The pure, testable helpers live in `agitrack/config/settings.py` (`normalize_menu_key`, `menu_key_label_for`, `menu_key_bytes_for`, `menu_key_is_shift`, `host_is_vscode`, `detect_menu_key_conflict`, `suggest_menu_keys`); the `GlobalConfig` menu-key properties delegate to them. Backend-agnostic.
- **Startup privacy warning wraps to the terminal.** The acknowledgment warning is stored as one flowing paragraph (`_PRIVACY_WARNING_TEXT`) and rendered through `_privacy_warning(width)`, which wraps at the authored width (`_PRIVACY_WARNING_WIDTH`, 73) on a normal/wide terminal but **re-wraps at the terminal's actual width when it is narrower**, so the line breaks land differently and the text never overflows or is chopped mid-word on a small terminal. Width defaults to `shutil.get_terminal_size().columns`; a leading blank line is preserved. On wide terminals the line breaks are identical to the previous hand-wrapped message.

## Backend Differences (gotchas)

**Claude and OpenCode have genuinely different mechanisms** (CLI flags, session storage, resume semantics, self-update, headless I/O). A feature that works on one can silently no-op or break on the other — most of the gotchas below were exactly that. So **every feature must work on BOTH backends**, and that must be **verified by actually invoking each backend with the feature** (a real `claude`/`opencode` run, not just unit tests with a mocked backend) before considering it done. Unit tests that stub the backend prove the wiring, not that the real CLI behaves as assumed.

Non-obvious places where Claude and OpenCode diverge. Each has caused a real bug; keep them in mind when touching either backend.

- **System prompt / bare (summarizer) runs.** Claude takes the summarizer's task instruction via `--system-prompt` and strips tools/memory/MCP for bare runs. **OpenCode `run` ignores `system_prompt`** (no flag to replace its system prompt or drop tools), so the instruction must be **folded into the prompt positional** for OpenCode — otherwise it never learns to summarize and the result is rejected by the echo/usability guards.
- **Resume working directory.** OpenCode records a per-session `directory` and on `--session <id>` resume **restores that recorded directory, ignoring the launch path / positional `project` arg**. A session whose dir moved (repo renamed, old worktree) resumes *outside* the worktree. Fix: retarget the session's recorded dir before spawn — `OpenCodeProxyAgent.retarget_working_dir` exports (unsanitized) and re-imports with the worktree as cwd; `opencode import` retargets dir to the import cwd and **overwrites the same id (no duplicate)**. Claude instead records cwd in its transcript (`retarget_session_cwd`). **Both resume paths must stage**: the multi-session path stages via `_new_session`, and the startup resume (`run()` → `_spawn()`) must call `_stage_backend_resume` itself, or the most common "resume my previous session" case is missed.
- **Self-update on macOS.** A Homebrew-installed OpenCode self-updates via `brew upgrade`, which runs Homebrew's own `sandbox-exec` — and **macOS forbids nesting `sandbox-exec`**, so it can never work inside aGiTrack's sandbox. There is **no env escape** (no `HOMEBREW_NO_SANDBOX` for macOS formula builds; `HOMEBREW_AVOID_NESTED_SANDBOXING` hard-errors at the default `/opt/homebrew` prefix). aGiTrack therefore drives the update itself from its **unconfined proxy** for a brew-managed backend on macOS — **regardless of the sandbox toggle** (sandboxed: the backend can't; unsandboxed: it only nags with a prompt), and sets `OPENCODE_DISABLE_AUTOUPDATE` on the backend child to suppress OpenCode's own (redundant/failing) prompt. Claude's native install self-updates fine (writes to `~/.claude`, no `sandbox-exec`).
- **Update command quirks.** `claude update` / `opencode upgrade`. `opencode upgrade` **exits 0 even when its brew step failed** — judge success by comparing CLI version before/after, not the exit code. Don't pre-gate on `brew outdated`: its local tap is stale until `brew update`, and it prints the **tap-qualified** name (e.g. `anomalyco/tap/opencode`, not bare `opencode`).
- **Session ids & store.** OpenCode assigns its own ids (`new_session_id()` → None; discovered after the run via a pre-spawn snapshot; ids are `ses_`-prefixed) and keeps sessions in a global DB (not plain files): `opencode export <id>` is global by id, `opencode session list --format json` is global (aGiTrack filters by `directory`), `opencode session delete <id>` exists. Claude forks a **new id on `--resume`**.
- **Headless process model.** OpenCode `run` streams JSON events via `Popen` and reads to EOF (a plain `timeout=` won't interrupt a stalled read — needs a watchdog that kills the process). Claude uses a blocking `subprocess.run` (add `timeout=` directly). Both summarizer paths need a timeout, or a hung call wedges the per-session one-at-a-time summary queue.

## Adding a New Backend (checklist)

A backend is more than a CLI wrapper — it shares the host terminal and the reactor with whatever ran before it, and **switching backends at runtime is a first-class case** (Ctrl-O → agent-backend). Most subtle bugs come from state or terminal modes that belong to one backend leaking into the next. Work through this list; verify on a real run of **every** existing backend plus the new one, proxy AND headless, on each platform touched (a mocked-backend unit test proves wiring, not behavior).

**State ownership — per-session vs. shared.** This is the rule that prevents cross-backend leakage:

- **Per-session (auto-isolated): put it in `Session.FIELDS` (`session.py`), with a default in `Session.bare()`.** Every per-turn / per-input / per-view field (`agent_in_flight`, the parse/commit trackers, `passthrough_prompt`/`passthrough_escape`, `pending_forwarded`/`pending_prompt_text`, `child_mouse`, `scroll_back`, `last_status`, selection state, …) lives there and is read as `runner.X` through the property layer. `_new_session` does `self.active = Session.bare()` and `_switch_active` swaps `self.active`, so any FIELDS state starts clean for the incoming backend automatically. **Never add new per-backend state as a bare `self.X` on the runner** — that's process-global and WILL bleed across a switch. (`_restart_agent`, the no-worktree restart-in-place path, additionally calls `_reset_agent_tracking` because it reuses the same `Session` rather than creating a bare one.)
- **Shared / process-global (the real carryover surface): the host terminal.** There is one console for the whole process, so its state is NOT per-session and a switch does not reset it for free:
  - **Raw/cooked mode, code pages, the saved-mode snapshot** — `_winconsole.RawConsole`. Its `enter()`/`leave()` are idempotent on purpose (see Native Windows); don't reintroduce per-call flushing/re-saving.
  - **Mouse reporting + ownership.** aGiTrack renders the screen itself (pyte), so the host never sees a backend's mode switches unless `_sync_terminal_modes` mirrors them, and `_enable_host_mouse` (re-)asserts host wheel reporting on spawn/switch. `child_mouse` (per-session, detected from the backend's `?1000/1002/1003h` output) decides whether a wheel event is forwarded to the backend or kept by aGiTrack for scrollback. A new backend that uses mouse differently (or not at all) must be handled in `_sync_terminal_modes`; if a switch leaves the host asserting mouse the new backend didn't ask for, its codes can leak into the prompt as literal text. **On Windows aGiTrack forwards all mouse (button/wheel/click/motion) + focus to the backend, the same as POSIX** — full pass-through, no Windows special-casing in `_sync_terminal_modes`. Motion (1002/1003) is mirrored too: the backend needs it for drag-select / copy. A deterministic ConPTY harness (`devtools/winmouse/`, run via `Start-Process` — the non-interactive agent shells can't give a ConPTY child a real console) proved conhost passes clean SGR mouse and focus events through to a VT-input backend verbatim, and end-to-end injection into live claude/opencode confirmed no leak; **a single backend works fully (mouse, scroll, copy).** **Neither backend uses win32-input-mode** — the `\x1b[?9001h` in their output is conhost's unconditional startup emission, not the backend, so encoding input as win32-input-mode would fix nothing. See `devtools/winmouse/FINDINGS.md`. **Open issue:** *switching* backends (e.g. Claude→OpenCode) leaks mouse codes and corrupts keyboard input (phantom char on Enter, Ctrl-C stops working) — process-global host/terminal state carried over from the first backend that the second doesn't reset; under investigation (a single backend is unaffected). On exit, `_finalize_pending_work` disables host terminal modes up front so a scroll during the (cooked, stdin-unserviced) "Finalizing…" teardown can't echo its raw report.
  - **Mirrored DEC modes** generally — bracketed paste (`?2004`), alt-screen (`?1049`), synchronized output (`?2026`), kitty keyboard — are tracked in `_sync_terminal_modes`. A new backend's mode requests must be recognized there or the host and the backend disagree about terminal state after a switch.

**Launch & process model.**

- Resolve the executable through `which_executable` (Windows: PATHEXT-aware, `.exe`/`.cmd`/`.bat`/`.com` only); a `.cmd`/`.bat` npm shim must run via `cmd.exe /c` for ConPTY. See the four `_conpty.py` invariants in Native Windows — all four broke a real launch.
- **No Ctrl-C may reach the backend during startup.** A false backend-exit (e.g. the reader thread raising on normal output) triggers a relaunch whose teardown `interrupt()` sends Ctrl-C; the result is the "exited repeatedly on launch" loop. The reader must never raise on well-formed output.
- Headless: give the run a real timeout/watchdog (see Backend Differences); `system_prompt` handling, session-id assignment, and resume-cwd semantics differ per backend (also above).

**Input handling caveat (open issue).** The wait-with-cancel loops (`_read_stdin(32)` while a fetch/parse/export thread runs) read stdin only to detect Ctrl-C/Esc and **discard** anything else, so keystrokes typed while such a wait is active are dropped rather than forwarded to the backend. This is the **suspected** cause of an observed bug: the first burst of keystrokes typed right after a Claude→OpenCode switch in worktree mode is lost (`switchtest…` → `s     st…`), and adding any per-character latency makes it disappear — i.e. it's a throughput race against a background operation, not a state-carryover bug (session state is correctly reset). The fix is to buffer non-cancel input and replay it to the backend once the wait ends rather than discarding it; it still needs a reliable repro to confirm the exact loop and verify the fix.

## Session Tracking

- aGiTrack tracks exactly one backend session per repository and pins to the session it launched, rather than chasing whichever session is globally newest.
- For backends that accept an explicit session id (Claude), aGiTrack pins at launch. For backends that assign their own id (OpenCode), aGiTrack snapshots existing session ids before launch and adopts the newly created one on the first parse, then stays pinned to it.
- **Following an in-backend switch (all modes).** When the user starts/resumes a different conversation *inside* the backend, aGiTrack follows it: worktree mode via `backend.latest_session_id` (the worktree dir is unique to the session), no-worktree mode via snapshot-based discovery (`_discover_spawned_session`, which only ever returns a session that appeared after launch, so preferring it over the pinned id is always safe). Combined with the per-conversation watermark (see the token exactness note), switching back and forth counts each conversation's turns exactly once. Background mode uses `latest_session_id` to track whichever session the user is driving.
- The `session` command (proxy) lets the user start a new session, switch the tracked session to another existing one, or sync tracking to the most recently active session (useful after starting a new session inside the backend TUI). Switching or starting a new session relaunches the backend TUI and re-baselines so existing history is not re-committed.
- Session detection, listing, and switching must work identically for both OpenCode and Claude.
- Shared-session conflicts (two contributors pushing the same session concurrently) are resolved by a best-effort **union merge** of the diverged copies rather than silent last-write-wins: `merge_transcripts` keeps the shared prefix and appends each side's unique rows by id (Claude JSONL; OpenCode's single-object export isn't line-mergeable, so it falls back to last-write-wins). **Before a merged transcript is uploaded** it must pass `_transcript_is_readable(text, backend)` — the backend's own parser (`parse_rows` / `parse_exported_session`) must still yield ≥1 turn — or the merge is discarded and last-write-wins is used, so a corrupt merge is never pushed. This runs in `store._add_and_push` before the push, for both backends.
- **Sharing onto a newer shared copy (overwrite).** The recency guard (`_would_regress`) normally refuses to replace a longer shared copy with a shorter one (`PublishResult.behind=True`) so a stale machine can't rewind everyone's copy. When a *manual* share is refused this way, the user is asked whether to overwrite it: `_share_session`'s background outcome stashes the refusal in `_pending_share_conflicts`, and `_service_share_conflicts` (a main-loop tick run AFTER `_service_background_share_ops`, so it never mutates that list mid-iteration) pops a prompt via `_resolve_share_behind`. Choosing overwrite re-shares with `store.publish(..., overwrite=True)`, which syncs the remote first (so the lease matches), then **skips both the union merge and the recency guard** to replace the shared copy wholesale. Only overwrite-or-keep is offered, by design: copies that CAN be combined are union-merged automatically during a normal publish (the superset never trips `behind`), so the only sessions reaching this refusal are non-mergeable (OpenCode) ones where replace-or-keep is the real choice. Auto-share never overwrites — it stays conservative and just refuses to regress.

## Concurrency and Locking

- Only one aGiTrack process may auto-commit/merge in a given working tree at a time. A process acquires a single-writer lock at `<tree>/.agitrack/lock`, released the instant the owner dies: an `fcntl.flock` on POSIX, an `msvcrt.locking` byte-range lock on native Windows (`agitrack/git/lock.py` picks the primitive by `sys.platform`; the Windows lock sits on a byte far past the pid JSON so a reader — e.g. the VS Code extension — can still read it). A second aGiTrack process on the same repo **refuses to start**: it prints `already_running_message` (which names the holding PID and tells the user to stop it first) and exits non-zero. Wired into both proxy and JSON modes.
- Quitting a proxy instance asks for confirmation before exiting.
- Concurrent sessions are isolated with git worktrees so changes are never attributed to the wrong session; see Concurrent Sessions below.
- **Idle low-power loop (battery).** The reactor's `select` timeout is set by `_select_timeout()` to "time until the next thing must happen", never a fixed busy-poll: `ACTIVE_POLL_SECONDS` (a coarse **1s** — every background sweep already self-throttles to its own interval, ≥2s, so polling faster buys nothing) while there is work to service, backing off to `IDLE_POLL_SECONDS` (`timings.idle_poll_seconds`, default 30s) once `_is_idle()` is true — no turn in flight (active or background), no pending per-turn pipeline work (file-change event, parse/commit, deferred Enter/queued prompt), no live worker thread, no timed notice on screen, and no keystroke or backend output for `timings.idle_after_seconds` (default 30s). Two sub-second exceptions: a queued repaint flushes at `0.016` (the coalesced-output frame), and a deferred prompt-submit (`_pending_enter_at`, scheduled 0.4s out) collapses the timeout to its exact remaining time so the injected Enter isn't stretched to a full second. `select` watches stdin **and every session's PTY** (active + all background fds), so it returns the instant the user types or **any** session — foreground or background — emits output; idle ends with no lag and commit/integration timers resume immediately. Waking is unavoidable for readable fds, but only a **real keystroke** resets the idle backoff: `_reactor_stdin_phase` gates `last_user_input` on `_is_real_keypress(data)`, so a mouse wheel / move / click or a focus report wakes us just long enough to redraw the scrolled view and then drops straight back to the low-power cadence (passive reading needs no commits or polling). Crucially, `_is_idle()` only goes true 30s after the last keystroke AND once output is quiet AND nothing is pending, so during active back-and-forth work aGiTrack never idles and every aGiTrack-triggered commit/integration runs at the active cadence — the long idle block only happens when the user has genuinely stepped away. Every deferred-commit/integration path that waits (parse thread, summary worker, `parse_pending`/`file_change_event`, background-session debounce) is itself a non-idle condition in `_is_idle()`, so a pending commit can never get stranded behind the 30s block. A sticky message (one that waits for a keypress) does **not** keep the fast loop, since the dismissing keypress wakes `select` on its own. This stops the process waking several times a second when nothing is happening — which is what made the host editor (e.g. VS Code) report aGiTrack as a significant energy user.
- **File/folder checks stay cheap.** Change detection is event-driven (watchdog `Observer`, native FSEvents/inotify) — never a tree walk; the `Observer is None` fallback is an interval-gated `git status_short()`. Session discovery globs `*.jsonl` and orders by `stat().st_mtime` (no file reads), and a session's label reads only the head of its transcript (`_session_label`, 100-line cap with an early break). Per-loop sweeps each gate on their own `*_SECONDS` interval before shelling out to git, so the 1s active poll never spawns a check more than its interval allows. The base-repo polls also **mtime-gate** their git subprocess: `_check_base_branch_drift` only runs `current_branch()` when `.git/HEAD`'s mtime moved (a checkout — not a commit), and `_poll_base_advanced` only runs `rev-parse` when the base ref / `packed-refs` mtime moved (`_newest_mtime`/`_base_git_dir`), so a steady repo costs a couple of `stat()`s per tick, not a `git` fork.
- **The git pipeline runs on a dedicated worker thread — never the main thread.** The foreground session's automatic `git status`/commit/merge runs in `_git_worker` (`_git_worker_loop` → `_run_git_pipeline`, started in `run()`), so a git subprocess can **never** block the reactor thread that echoes the user's keystrokes — typing is always instant, even right after an edit. The worker sleeps on `_git_wake` (set by the file watcher on a worktree write, and once per reactor tick) and backs off to the idle cadence via `_is_idle()`. **Marshaling rule:** the worker must never touch stdin or the screen. Any dialog it raises (merge-conflict resolve, copy-file/untracked offers) goes through `_run_modal`, which when off the main thread enqueues an `_ModalRequest` on `_modal_mailbox` and blocks; the main thread presents it inline (`_drain_modal_mailbox` → `_run_modal_inline`) and returns the answer. `_render` off-thread just sets `_render_pending` (the main loop flushes it); `_set_message` only writes fields, so it is thread-safe as-is. A self-pipe (`_wake_r`/`_wake_w`, in the `select` set) lets the worker wake the reactor at once to present a dialog. PTY writes are serialized by `BackendProcess._write_lock` so a worker-injected conflict prompt can't interleave bytes with a forwarded keystroke.
- **`self.active` is single-threaded; the worker never swaps it.** The whole runner keys per-session state off one `self.active` pointer that `_with_session` temporarily swaps. To stay safe with a concurrent worker, the worker's pass operates **only on the foreground session and never swaps `self.active`** (verified: `_maybe_agent_commit`/`_maybe_complete_agent_merge`/base polls reach no `_with_session`/`_switch_active`). Background / multi-session servicing — which *does* swap `self.active` (`_service_background_sessions`, the background loops of `_service_commit_summaries`/`_sync_idle_worktrees_to_base`, and `_check_base_branch_drift`'s dir-change path, plus `_ensure_worktree_alive`'s PTY/watcher work) — **stays on the main thread**, inside `_reactor_timers_phase` under a **non-blocking** `_pipeline_lock.acquire(blocking=False)` (skip-this-tick if the worker holds it, so the reactor never stalls). The worker holds `_pipeline_lock` across its pass; user-initiated foreground swaps (`_switch_active`) and exit acquire it via `_acquire_pipeline_lock_from_main`, which **drains the modal mailbox while it waits** so a worker blocked on a dialog can't deadlock the acquirer. Exit/restart calls `_stop_git_worker` (joins, cancelling any pending dialog) *before* `_finalize_pending_work`, so exit-time commits are the only git that runs on the main thread — teardown, never the live path. Scope note: prompt-submit-time git (`_integrate_committed_turn_before_new_turn`/`_ensure_turn_branch` on Enter) and user-initiated commands (`git-user-commit`) stay inline on the main thread — they are at a keypress the user is waiting on, not the typing hot path.

## Native Windows (#118)

- aGiTrack runs natively on Windows (PowerShell / Windows Terminal), not only via WSL. The POSIX-only mechanics sit behind a platform layer so the ~9,400-line reactor stays platform-agnostic; macOS/Linux keep their exact existing path.
- **Platform package `agitrack/proxy/platform/`.** `base.py` defines the `ChildProcess` / `HostTerminal` / `ReactorWaker` Protocols both platforms satisfy; `make_child_process` / `make_host_terminal` / `make_waker` (in `__init__`) dispatch on `os.name`. `posix.py` thinly wraps today's `BackendProcess`/`TerminalHost`/`os.pipe`; `nt.py` + `_winconsole.py` are the Windows implementations (imported only on Windows).
- **The socket-bridge insight.** Windows `select()` accepts **sockets** but not console/pipe/ConPTY handles, and `socket.socketpair()` works on Windows. So the reactor keeps its `select`-based loop: every native input source (the backend's ConPTY output, host-console stdin, the resize watcher, the worker waker) is bridged through a reader thread into one end of a `socketpair`, and the reactor selects on the other end. The fd vocabulary (`master_fd`, `stdin_fileno`, `wake_fileno`) is preserved, so the reactor changed by only a handful of identifiers (`_stdin_fileno()`/`_read_stdin()` helpers; `make_waker()`).
- **`NtChildProcess`** spawns the backend under a ConPTY using **our own raw `CreatePseudoConsole` driver (`_conpty.py`), NOT `pywinpty`** — pywinpty's ConPTY backend kills every child with `STATUS_CONTROL_C_EXIT` (`0xC000013A`) when the app is PyInstaller-frozen (the MSI), so no backend can launch from the MSI. A daemon reader copies the pseudoconsole output into the bridge socket (same back-pressure as the POSIX PTY). `_resolve_windows_command` resolves the backend via `which_executable` (PATH/PATHEXT-aware; returns only a runnable `.exe`/`.cmd`/`.bat`/`.com`, never a `.ps1` or extensionless shell shim) and wraps a `.cmd`/`.bat` (npm's `claude.cmd`) through `cmd.exe /c` — the only way ConPTY's `CreateProcess` runs a batch shim — so **both** backends launch. `interrupt()` writes ETX (Ctrl-C) to the ConPTY; `poll()` reports the child gone only after the reader thread has finished draining (so a fast child's trailing output isn't lost). **Four non-obvious `_conpty.py` invariants, each of which broke a real backend launch — keep them:** (1) read into an **unsigned `c_ubyte`** buffer — a signed `c_byte` array indexes to −128..127, so `bytes(buf[:n])` raises `bytes must be in range(0, 256)` the instant the backend emits any byte ≥ 128 (its UTF-8 TUI), which the reader catches as "child gone" → false exit → relaunch → teardown's `interrupt()` Ctrl-Cs the backend → "exited repeatedly on launch"; (2) pass the `HPCON` to `UpdateProcThreadAttribute` **by value**, not `byref` — `byref` stores the handle's address as the pseudoconsole, so the child never attaches and its output goes to the parent console (interactive: handed off to a new Windows Terminal window; headless: lost); (3) creation flags are **`EXTENDED_STARTUPINFO_PRESENT` only — do NOT add `CREATE_NO_WINDOW`**, which also breaks the attachment; (4) `bInheritHandles=FALSE` so the child takes its std handles from the pseudoconsole, not the parent's (otherwise stdout isn't a TTY and the backend drops to non-interactive mode).
- **`NtHostTerminal` + `_winconsole`** drive the Win32 console: VT raw mode via `SetConsoleMode` (`ENABLE_VIRTUAL_TERMINAL_INPUT`/`PROCESSING`, code pages set to UTF-8 / 65001 so box-drawing renders), a `ReadFile`-on-the-console reader thread feeding the stdin bridge, `GetConsoleScreenBufferInfo` for size, and a polling **resize watcher** that replaces `SIGWINCH` via `consume_resize_pending()` in the select phase. SIGWINCH/SIGHUP installs are guarded off on Windows. **`RawConsole.enter()`/`leave()` are idempotent** (guarded by an `_entered` flag): `set_raw()` can be called again while already raw (a render/resume that didn't pass through `set_cooked`), and a naive re-entry would (a) re-save the already-raw modes over the real cooked ones so `leave()` could never restore the terminal (garbled output, notably over RDP after a turn), and (b) re-`FlushConsoleInputBuffer` and discard keystrokes the user just typed. The one-time flush on a genuine cooked→raw entry is still correct (it drops input typed into a cooked-mode prompt, including a stray Ctrl-C that would otherwise launch-kill the backend).
- **Process primitives `agitrack/proc.py`** (`detach_kwargs`/`pid_alive`/`terminate_pid`) abstract `start_new_session`/`os.kill` for the dashboard daemon and self-update (`CREATE_NEW_PROCESS_GROUP|DETACHED_PROCESS|CREATE_NO_WINDOW` + Win32 `OpenProcess`/`TerminateProcess` on Windows). The repo lock uses `msvcrt` (see Concurrency). OpenCode's transcript export falls back from `pty.fork` to a `subprocess` pipe on Windows.
- **Graceful shutdown.** A closing VS Code window can't deliver a catchable signal to aGiTrack on Windows, so the extension writes `<repo>/.agitrack/shutdown`; the reactor polls it (`_check_shutdown_request`, ~1 s, in the timers phase), finalizes the turn, and exits — the cross-platform equivalent of SIGTERM (the extension also `taskkill`s as a last resort). **Confinement where the OS sandbox is unavailable (Windows, or POSIX without `bwrap`/`sandbox-exec`).** No OS filesystem sandbox, so two prompt-independent mechanisms take over, both gated on `_base_commit_guard_active()` (worktree mode + confinement on + `not sandbox.is_available()`): (1) a **base-repo `pre-commit` guard hook** (`agitrack/git/hooks.py`, installed by `ProxyRunner._install_base_commit_guard`, removed on teardown) that **hard-blocks the agent from committing into the base repo**; and (2) `_warn_if_base_edited`, which warns about uncommitted base writes. The hook attributes reliably — the only way to, since agent and user share one git identity and the user can edit/commit at any time — via the `AGITRACK_WORKTREE_GUARD` env marker set in `_backend_child_env` ONLY on the agent child in worktree mode: every `git` the agent runs inherits it, while the user's shell, agents run outside aGiTrack, and `--no-worktree` agents don't have it, so they commit freely (and a hook left behind by a crash is a no-op for them). Commits inside a linked worktree are allowed; only base/main-repo commits are rejected (detected via `git rev-parse --absolute-git-dir` containing `/worktrees/`). It chains any pre-existing `pre-commit` hook (moved to `pre-commit.agitrack-orig`, restored on removal), is skipped when `core.hooksPath` is set, is bypassable with `git commit --no-verify`, and is disabled by turning confinement off (`--no-confine`/`--no-sandbox`/`sandbox: false`). A full *filesystem* sandbox (blocking arbitrary writes, not just commits) still isn't feasible for a *same-user* child on Windows — Job Objects govern CPU/memory not ACLs; AppContainer needs per-capability SIDs and breaks Node/`git`; a restricted-token + deny-ACE scheme is heavy/fragile — so on these platforms writes are *monitored* (warned) while commits are *prevented*.
- **VS Code extension** (`editors/vscode`): native Windows is un-gated; launch uses PowerShell-aware quoting + a `$LASTEXITCODE` exit suffix (not `&& exit`); liveness uses `Get-CimInstance Win32_Process` (not `pgrep`/`ps`); install-path resolution targets `agitrack.exe` in `Scripts`/pipx dirs.
- **CI**: a dedicated `windows-latest` job (`.github/workflows/ci.yml`) runs an import smoke + a curated, platform-relevant subset (`test_proc`/`test_dashboard_daemon`/`test_shutdown_request`) and a real ConPTY spawn/read check (`test_windows_conpty`, `skipif os.name != "nt"`). Windows-only branches are kept mypy-clean via `sys.platform`-narrowing and targeted `# type: ignore` (the POSIX gate `./scripts/check.sh` stays green).

## Concurrent Sessions (worktrees + auto-integration)

This is the design aGiTrack targets for running several sessions at once. Foundations (`agitrack/git/lock.py`, `agitrack/git/worktree.py`, and the worktree/branch/merge helpers in `agitrack/git/repo.py`) are implemented and unit-tested; the multiplexer wiring in `agitrack/proxy/runner.py` is the remaining integration.

- A single aGiTrack process multiplexes several live sessions; one is displayed, the others keep running and integrating in the background.
- The main working tree / base branch is mutated only by the serialized merge coordinator; every session runs in its own worktree under `.agitrack/worktrees/<name>`.
- A session creates a transient turn branch (`agitrack/<name>/t<n>`) when it receives a prompt; on the turn's final agent message the branch is integrated into the base and deleted.
- Integration is serialized and completion-ordered. It runs inside the owning session's worktree (merge base into the turn branch) so the agent can resolve conflicts in place; aGiTrack auto-prompts the agent with the conflicting commits' context and pauses for the user if it cannot resolve.
- A `session` view shows running/idle/merging status and can stop sessions; a `base` command switches the base branch after stopping sessions and draining pending integrations; on restart aGiTrack offers recovery for stale `agitrack/*` branches and worktrees.
- The `--delay-merge` flag (`ProxyRunner._delay_merge`, off by default) holds automatic integration: after a turn commits, the in-session auto-merge paths (`_integrate_session_turn`, `_integrate_committed_turn_before_new_turn`, `_integrate_agent_made_commits_if_idle`) defer and a notice names the **working directory** (the worktree path, which the user may not otherwise know) so they can review/edit. The merge runs only on explicit confirmation via the session menu ("Merge reviewed changes into &lt;base&gt;" → `_integrate_active_session`). Exit still finalizes (`_integrate_session_on_exit` is not gated) so committed work isn't stranded, and unmerged work is surfaced again next run.
- **A user-requested merge is best-effort — it must not dead-end on uncommitted changes.** `_integrate_active_session` only refuses while a turn is GENUINELY in flight (`_agent_is_active()`, after a `_clear_agent_in_flight_if_idle()` refresh so a finished-but-not-yet-cleared turn doesn't falsely block). Otherwise it commits the worktree's committable leftovers itself (`_commit_latest_turn_sync`) and proceeds; anything still uncommitted afterward is non-committable (git-ignored / intentionally-unstaged) and simply stays in the worktree rather than aborting the merge of the committed work. The commit-before-merge lives in this one place; `_merge_active_into` just retargets the base branch and delegates here. (Earlier the guard refused on any `repo.has_changes()`, stranding the user with a misleading "finish or stop the current turn" message even when no turn was running.)
- **A session's worktree must never be stranded as an "open end" by uncommitted leftover files.** On exit (`_remove_worktree_on_exit`) a worktree is kept only for a genuine reason to preserve work: an in-progress merge (`merge_ctx` / `merge_in_progress()`), or commits still ahead of the base (`log_range`). Uncommitted leftover FILES (untracked, git-ignored, or files the user declined to stage) do NOT keep it — they are what the copy offer handles, and since a copy leaves the source in place, refusing removal on `repo.has_changes()` left dirty worktrees lingering forever even after the user copied the files out. On an interactive exit (`screen is not None`) the copy offer runs (`_offer_copy_unstaged_to_base(context="exit")`, which warns that declining discards them) and the worktree is then force-removed (`WorktreeManager.remove` already uses `force=True` with a `shutil.rmtree` fallback). The single exception is a **signal teardown** (terminal/window closed, `screen is None`): with no UI to rescue the files, a worktree that still has leftover files is KEPT so the next startup surfaces them instead of silently discarding the user's work.

## Backend Selection and Global Config

- The selected backend is stored per repository in `.agitrack/state.json`.
- A user-wide config at `~/.agitrack/config.json` (override the directory with `AGITRACK_CONFIG_DIR`) stores `default_backend`, used when a repository has no backend recorded yet.
- **Settings precedence + menu.** `GlobalConfig` carries a repo-local overlay loaded from `<repo>/.agitrack/config.json` (`load_repo_overlay`), so the effective value of any setting is **repo-local > global > built-in default** (every getter reads through `_raw`). `source(key)` reports which layer a value comes from; `set(key, value, scope=…)` / `unset(...)` write the global file or the repo file (`save_repo` preserves other keys the repo file holds, e.g. AgitrackState's). The `settings` Ctrl-G command (`ProxyRunner._settings_menu`, a declarative `_settings_specs` registry) edits **all** options — backend, sandbox, allowed_edit_paths, use_worktrees, commit_guidance, summarization, check_for_updates, menu_key, and a timings sub-menu. It is a **form**: each edit picks its own scope ("Apply to: This repository / Global") and is held as a **pending** change in `_settings_pending` / `_settings_pending_timings` (shown as `· UNSAVED → scope`); pending edits are written only when the user **confirms save on close** (`_confirm_save_pending` → *Yes, save / No, discard / ← Keep editing*). Launch-resolved settings (sandbox, allowed_edit_paths, worktrees, commit_guidance, backend, menu key, timings) note "won't take effect until YOU restart"; nothing is live-applied to the running process.
- **Menu navigation: Esc = up one level, everywhere.** Every Ctrl-G menu is one `while True` loop returning just **two** signals: `_MENU_UP` (Esc/back → the caller shows itself one level up) and `_MENU_DONE` (a context transition — switch/new/resume a session — unwinds to the agent). Authoring rule: build options → `_select_popup` → dispatch; Esc → `return self._MENU_UP`; a transition → `return self._MENU_DONE`; any other action (config, or a child menu the user backed out of) → `continue` (re-show this menu). A child menu is entered by **calling** it: `if child() == self._MENU_DONE: return self._MENU_DONE` else `continue`, so the child's Esc lands back on this parent while a transition unwinds past every level. `_after_menu_command` turns a top-level menu's `_MENU_UP` into "re-open the Ctrl-G palette" (the parent of every command menu). Navigation is **silent** (no "closed"/"cancelled" message on the way up) and modal closes don't repaint the bare screen (`_run_modal` flags `_render_pending` instead of `_render()`), so stepping between levels is instant and flicker-free. Parent→child: `_run_command` → {`_session_menu`, `_settings_menu`, the summarizer menu, backend picker}; `_session_menu` → {`_rename_session_menu`, `_change_session_merge_branch_menu`, `_share_session`, `_manage_shared_sessions_menu`, `_resume_session_menu`, `_resume_shared_session_menu`, `_prompt_new_session`, …}; `_manage_shared_sessions_menu` → `_manage_one_shared_session`. The convention is documented in a comment above `_run_command`.
- **Popups wrap and scroll to fit any terminal.** `renderer._wrap_markup` and `textwrap` break even a long unbroken token (e.g. a path) so it never overflows the box border. `SelectModal` windows BOTH its `detail` list and its `options` list (keeping the selected row visible, with ↑/↓ "more" hints) so a tall menu scrolls instead of being truncated. `append_box(scrollable=True, scroll=…)` windows an over-tall popup **message**; the runner tracks `_message_scroll`/`_message_max_scroll`, resets scroll in `_set_message`, and `_reactor_stdin_phase` routes PgUp/PgDn to message scrolling (refreshing its expiry) whenever a message overflows the screen.
- On the first run (no global default yet and no `--backend`), `select_default_backend` lists the backends alphabetically with each one's install status and **asks whether to install any of the not-installed ones** — a single one (by number) or all of them (`all`). Already-installed backends are shown as installed and never offered for reinstall, and skipping is always allowed. The default becomes the first installed backend (or the first listed when none is installed yet — the launch gate then offers to install it).
- **Automatic backend install (`install_backend`, opt-in).** When the user chooses to install a backend — at first-run or at the launch gate (`ensure_installed_backend`, "Press Enter to install … now") — aGiTrack installs ONLY that one backend, cross-platform: the backend's official install script on macOS/Linux (`curl … | bash`, self-contained, no Node), npm everywhere (`npm install -g …`), and a **winget Node bootstrap** on Windows when npm is absent. Crucially, the freshly-installed CLI's directory is added to **this process's** `PATH` (`_add_dirs_to_path` over `_candidate_bin_dirs` — npm's global prefix from `npm prefix -g`, `~/.local/bin`/`~/.opencode/bin` on POSIX, `%APPDATA%\npm` + the Node dir on Windows) so the backend is runnable immediately, with no aGiTrack restart — the OS installers only update the registry/profile PATH that *new* shells inherit. `install_backend` is dependency-injected (`run`/`which`/`output_fn`) so it is fully unit-tested on POSIX, and never raises. This matters most for the MSI install path: a fresh Windows machine has aGiTrack but no agent, and aGiTrack sets one up on first run.
- Check that the backend's CLI is installed (on `PATH`) before launching it (`ensure_installed_backend`). If it is not, offer to install it automatically, switch to a different (installed) backend, or show manual instructions; backend switching is likewise blocked for backends that are not installed.
- **System prerequisites — git, gh, and git identity (`agitrack/system_tools.py` + cli.py).** On an interactive launch aGiTrack also helps set up its non-backend prerequisites. `_maybe_install_tool` (cli.py) offers to install missing **git** (required, gates launch) or **gh** (optional, in `_check_gh_availability`) via `install_system_tool`, which uses the platform package manager — winget on Windows, Homebrew/`xcode-select` on macOS, the detected distro manager with `sudo` on Linux — and PATH-augments in-process like the backend installer. `_ensure_git_identity` (cli.py, proxy + TTY only) prompts for a missing global `user.name`/`user.email` and sets them, because git refuses to commit without them and aGiTrack commits every turn. gh sign-in is offered inline (`gh auth login`) when gh is installed but unauthenticated. All of this is TTY-gated and skipped for scripted/json runs (no prompts, no polluted machine-readable output).
- `agitrack --backend <claude|opencode>` selects the backend for a run and saves it as the new global default.
- Switching backends saves the current backend's session id and restores the target backend's last session id for the repository, so each backend keeps its own conversation.

## Self-Update

- aGiTrack can update itself in place. It detects how it was installed: **source-linked** (importable from a git checkout of its own source — the editable `pip install -e .`) or **package** (installed as a wheel). Implemented in `agitrack/update/`.
- Source-linked: compare the current branch against its upstream remote branch (after a `git fetch`); an update is available when the local branch is behind. Package: compare the installed version against the latest published one (`pip index versions`, run through the resolved pip invocation; reading the index is safe even on an externally-managed Python).
- Check for updates at startup and periodically while running (every `timings.update_check_seconds`, default 300s). The periodic check runs on a worker thread so the terminal never blocks on network I/O.
- When an update is available, prompt the user — at startup (before launching) and during a session (a status-bar notice plus the `update` command in the `Ctrl-G` menu).
- Never surface an update prompt while a merge / conflict resolution is in progress in any session (active or a background session integrating its turn); the notice is held until the merge is done, and an accepted update is likewise deferred.
- If the user accepts during a session, apply the update only once **every session has finished and all commits are integrated** (no agent in flight, no pending parse/merge/summary, no running background session), then re-exec aGiTrack (`restart_agitrack` → `_restart_command`) so the new code loads. The re-exec runs `python -m agitrack` for a normal install, but **a frozen MSI build re-runs `agitrack.exe` directly** — `-m agitrack` is not a valid argument to a PyInstaller executable, so passing it (the old behaviour) broke both self-update restart *and* the settings-menu "restart now" on MSI installs. `_restart_command` branches on `getattr(sys, "frozen", False)`; `tests/test_updater.py` locks both shapes.
- Source updates are fast-forward only and abort (with a message) when the checkout is dirty or has diverged from upstream, so an automatic update never clobbers local development.
- Package updates are deliberately **package-manager-independent** wherever possible: the primary path upgrades with the *running interpreter's own* pip (`<python> -m pip install --upgrade`), which upgrades a plain `pip` install, a venv, a `--user` install, and a pipx venv identically — no need to detect or shell out to pipx/brew/apt. It falls back to a `pip3`/`pip` on `PATH` only when that interpreter has no `pip` module. The single case pip can't handle is an externally-managed (PEP 668) Python — Homebrew's or a distro's — where pip refuses by design; there aGiTrack defers to the owning system manager (`brew upgrade`, the only system manager that ships aGiTrack — distro managers don't carry it), and when no automatic path works it reports a **full enumeration** of every manual route (pip, pipx, Homebrew, and `--break-system-packages`). Never auto-add `--break-system-packages` (it can corrupt a system/Homebrew Python). Install-method detection keys off where the running code physically lives (resolved package path, `sys.prefix` backstop), checking the pipx-venv marker before the Homebrew prefix.
- Update checks are on by default; the user can turn them off via the "stop checking" choice or `check_for_updates: false` in `~/.agitrack/config.json`. Scripted/non-TTY runs never prompt.
- An interrupted update must never leave aGiTrack **uninstalled**. `pip install --upgrade` removes the old version before writing the new, so a kill between the two would strand the package. The upgrade subprocess (pip, and the `brew upgrade` fallback) therefore runs with `start_new_session=True` (`updater._apply_package`), so a terminal-close **SIGHUP** — the user quitting VS Code / closing the window mid-upgrade — is not delivered to it and it runs to completion. The in-session apply path is also protected: `_handle_exit_signal` **returns without tearing down while `self._update_applying`** is set, so a SIGHUP/SIGTERM during the apply can't unwind `subprocess.run` and kill the (now detached) pip child. (Root cause of an earlier "aGiTrack uninstalled itself after I quit VS Code" report: the startup `Update aGiTrack now? [Y]` upgrade was interrupted between uninstall and reinstall.)
- A failed update must never take aGiTrack down or block work. `Updater.apply()` never raises — any failure (subprocess timeout, OS error, pip/git crash, PEP 668 refusal) becomes an `error` status carrying manual-update instructions (`Updater.manual_update_instructions()`), and aGiTrack keeps running the current version. On a failure aGiTrack records the target version in `pending_manual_update` (global config): the next startup shows a **single** manual-update reminder (and clears it once aGiTrack is actually at/above that version), while the regular in-session update notice is suppressed for that version so the user isn't nagged repeatedly. The explicit Ctrl-G `update` command can still retry.
- Running aGiTrack inside the VSCode extension does not change any of this: the extension launches the real CLI in an integrated terminal (a real PTY), so the startup TTY-gated update offer and the Ctrl-G update both work, and the `python -m agitrack` re-exec lands cleanly in the same terminal.
- **VSCode extension version is locked to the CLI version.** `pyproject.toml` is the single source of truth; `scripts/sync_vscode_version.py` stamps `editors/vscode/package.json` (and `package-lock.json`) to match, enforced by `tests/test_version_sync.py`. `agitrack --version` (cheap, no repo/privacy side effects) lets the extension detect a CLI that has self-updated past it and prompt the user to update the extension.
- **Releases ship the backend CLI and the VSCode extension together.** `release-patch.yml` (on merge to main) and `release-tag.yml` (on a `v*` tag) run the version sync, commit it with the bump, build and publish the CLI to PyPI, create a GitHub Release, and — when the `VSCE_TOKEN` repo secret is set — publish the extension to the Marketplace at the matching version, so a CLI release and its extension move together. `release-patch.yml` picks the bump level from the merge commit / squash-PR title: `[major]` bumps major (minor+patch→0), `[minor]` bumps minor (patch→0), and neither marker is a patch bump (default); markers are case-insensitive and the title is read via env, never interpolated into the shell.
- **Windows MSI is a first-class, standalone install method.** A PyInstaller one-folder bundle (`installer/agitrack.spec` → `agitrack.exe` + `_internal/`) wrapped by a WiX **v5** perMachine MSI (`installer/agitrack.wxs`; `UpgradeCode` is the stable product identity and must NEVER change between releases). WiX v5 is required because the file harvest uses the `<Files>` element (added in v5; v4 had no `wix harvest`/`heat`); the wix tool and `WixToolset.UI.wixext` are pinned to the **same** version (a mismatch fails `wix build` with WIX0144), the `<ui:WixUI>` element needs the `xmlns:ui` namespace declared, `SourceDir` is passed as an absolute path so the `<Files>` glob is unambiguous, and the build passes `-arch x64` (the bundle is 64-bit; the WiX default x86 package's `ProgramFiles64Folder` doesn't resolve to `C:\Program Files`, so files would install to the wrong place — caught by the CI install smoke-test). It bundles its own Python (and its own `.dist-info` via `copy_metadata("agitrack")` in the spec, so `agitrack --version` reports the real version instead of the frozen-build `0.0.0` fallback), so it installs and runs `agitrack` on a machine with **no Python/pip at all**, adding `C:\Program Files\aGiTrack` to PATH and writing `HKLM\Software\aGiTrack\InstallDir`. The MSI is built and attached to the release **inside the release workflows themselves** — a `build-msi` job (`windows-latest`, `needs: release`) in both `release-patch.yml` (the normal merge-to-`main` path) and `release-tag.yml` (human `v*` tag) builds `agitrack-<version>-windows-x64.msi` and `gh release upload`s it. It must live in the same workflow run as the release, NOT a separate tag-triggered workflow: GitHub does not trigger workflows from a tag/release created with the default `GITHUB_TOKEN`, so the old standalone `release.yml` never once fired and no release ever got an MSI (the bug this fixed). The CI `build-msi` job (`ci.yml`, on every PR) guards the bundle against rot: it builds the MSI from the pyproject version and proves it installs by itself — silent `msiexec /i … /qn` perMachine install (the runner has an admin token), asserts the installed `agitrack.exe --version` equals the pyproject version, then a clean silent uninstall. The VSCode extension uses this same MSI as its **no-Python fallback**: `installAgitrack()` (`editors/vscode/src/extension.ts`) only reaches `installViaMsi()` on Windows when `planInstaller()` finds neither pipx nor pip, downloading the latest release's MSI asset (selection logic in the pure, unit-tested `editors/vscode/src/msi.ts`) and running `msiexec`. It also discovers a by-hand MSI install via `msiInstallCandidates()` (a GUI-launched VSCode doesn't inherit the PATH entry the MSI adds — issue #93).
- **MSI self-update** (the frozen Windows bundle updating itself in place; design in `docs/design/msi-self-update.md`, implements #124). `Updater._install_method()` returns `METHOD_MSI` when **both** `sys.frozen` and the `HKLM\Software\aGiTrack\InstallDir` registry key are present (a portable PyInstaller zip lacks the key → falls through to pip). `check()`/`apply()` route to `_check_msi`/`_apply_msi`: the check queries the **GitHub Releases API** (`/releases/latest`, repo from `AGITRACK_GH_REPO` or the `core-aix/agitrack` default — an MSI install has no source remote to read; 5-min + ETag in-process cache keeps under the anon rate limit) for the newest `agitrack-*-windows-x64.msi` asset and parses the version from its name; apply **downloads** it to `%TEMP%` (size-capped, sha256-verified when GitHub supplies a digest) and stashes `Updater.pending_msi_path` — it does **not** install (the MSI replaces the running `agitrack.exe`, so the install must happen after exit). The hand-off is in the runner: `_apply_update_and_restart` sets `_pending_msi_handoff` when a download is pending, and the teardown calls `_launch_msi_bootstrapper()` instead of `restart_agitrack`. To keep the relaunched agent at the **user's** integrity level (not admin), the hand-off uses two cooperating processes: an **elevated** bootstrapper (`installer/agitrack-update.cmd`, run via `proc.shell_execute_runas` → `ShellExecuteExW(lpVerb="runas")`, UAC) waits for our PID, runs `msiexec /i … /passive /norestart REINSTALL=ALL REINSTALLMODE=vomus`, and writes its exit code to a marker file; a **non-elevated** PowerShell relauncher (spawned detached before we exit) polls that marker and, on success, starts the new `agitrack.exe` with the saved args. Launch args survive the replace via `%LOCALAPPDATA%\aGiTrack\last-args.txt`, written each startup by `_write_msi_last_args` (frozen-Windows only). UAC declined / `runas` failure ⇒ `_launch_msi_bootstrapper` returns False, the teardown falls back to a normal re-exec of the **current** version, and `pending_manual_update` is set so the next startup reminds the user. The bootstrapper is committed (not part of the PyInstaller harvest) and installed into `INSTALLFOLDER` by the `UpdaterComponents` group in `agitrack.wxs`, so the same MSI that updates everything else updates the bootstrapper. Unit tests monkey-patch the GitHub HTTP, the registry read, and the download, so the whole path is covered on POSIX CI (`tests/test_updater.py`).

## Testing

- Make it regular practice to **create temporary git repositories** to test aGiTrack functionality end-to-end against real git, rather than relying only on mocked inputs. Where a feature depends on backend behavior (transcripts, sessions, sub-agents, token usage), also run the actual backend CLIs (`claude`, `opencode`) in a temp repo to produce real data and verify against it. This is the established practice for branch/worktree concurrency, sub-agent token accounting, and token-count precision.
- Compute expected results independently from the real data and assert **exact** equality — especially for token counts, which must be precise (not more, not less).
- Capture the verified scenarios as permanent tests under `tests/` so the behavior stays locked in, and clean up any temporary repositories afterwards.
- **User-flow test matrix (`tests/FLOW_MATRIX.md`) — keep it complete.** It is the auditable index of every interactive sequence a user can drive (startup, prompt submission, the agent turn, interruption/follow-ups, copy-back, user/agent commits, switching, backend switch, background sessions, integration/merge/conflict, exit, sharing, self-update, the Windows specifics, AND the system-prerequisite/installation flows — missing git/gh, git identity, gh login, backend install). **Any change that adds or alters a user flow MUST, in the same change: (1) add or extend the covering test, (2) add/update its row in `tests/FLOW_MATRIX.md`, and (3) if a sequence can't be covered yet, list it under that file's "Known gaps / TODO" rather than leaving it implicit.** A flow change is not "done" until it is represented there. Prefer **real-git** tests (a real temp repo) over mocks for anything touching commit/merge/worktree/copy — mocks that stub `repo.commit` cannot catch real `git` failures (this is how the Windows cp1252 commit-encoding bug slipped through).
- `./scripts/check.sh` (ruff check, ruff format, mypy-vs-baseline, pytest + coverage) must pass before a change is considered done.

## Diagnostics & Debugging (DEBUG_PROXY / DEBUG_RAW)

aGiTrack ships two always-present, opt-in diagnostic switches for debugging the proxy reactor — input/output, terminal-mode mirroring, session switches, hangs, and leaks. They are **off by default**, gated so they cost nothing in normal use, and behave **identically on Windows and macOS/Linux** (same env vars, same log files, same format) — so a bug report from either platform produces the same artifacts. Keep them; they are the first thing to reach for when an interactive glitch is reported.

- **`DEBUG_PROXY`** — human-readable event log. Records the stdin parse pipeline (each chunk: raw bytes → after menu-key intercept → after kitty decode → feed result with `forwarded`/`command`/`capturing`), backend spawn/switch/restart decisions, session lifecycle, agent-parse worker start/finish, shutdown, and a throttled (~5 s) **idle heartbeat** showing the loop is alive when no keystroke arrives. `agitrack --verbose` implies it.
- **`DEBUG_RAW`** — byte-exact I/O capture for replaying an interactive glitch (mouse/keyboard escape sequences, a backend's native picker). **Implies `DEBUG_PROXY`.** Every record is `HH:MM:SS.mmm <TAG> b'<repr>'`, flushed line-by-line (open+write+close per line) so the log survives a hard kill when a leak has broken Ctrl-C / graceful exit. Tags: `>` host→backend input (includes bytes read inside popups/modals), `<` backend→host output, `M` a terminal-mode sequence mirrored to the host console (mouse/kitty/paste — `_sync_terminal_modes`), `EOF` the backend's pty/ConPTY closed.

**Setting them.** Each switch is read under THREE names (first found wins): the bare name (`DEBUG_RAW`), the project-prefixed `AGITRACK_DEBUG_RAW`, and the legacy `AGIT_DEBUG_RAW` (see `agitrack/env.py` + `ProxyRunner._debug_flag`). Truthy values are `1` / `true` / `yes` (case-insensitive). The bare form exists because these are throwaway switches a user sets ad hoc in a shell — reaching for `DEBUG_RAW` is the obvious move, and a prefixed-only flag silently produced no log.
  - macOS/Linux (bash/zsh): `DEBUG_RAW=1 agitrack` (one run) or `export DEBUG_RAW=1` (the shell session).
  - Windows (PowerShell): `$env:DEBUG_RAW = "1"; agitrack` — set it in the **same** terminal that launches aGiTrack (the VS Code extension launches a child shell, so set it in that integrated terminal or in the OS environment).

**Where the logs go.** One file per run, in the **base** repo's `.agitrack/` (never the session worktree's — the worktree is removed on exit, which would destroy the log): `<base-repo>/.agitrack/proxy-debug-<YYYYMMDD-HHMMSS>.log` and `proxy-raw-<YYYYMMDD-HHMMSS>.log` (`_diag_path`; the `<YYYYMMDD-HHMMSS>` run-stamp is shared by both files of a run). To inspect the latest run, sort that directory by name/mtime and read the newest `proxy-*` pair; share both when filing a report.

**Cross-platform parity / the one platform-specific detail.** The flags, both log files, and every record format are the same on all platforms. The only platform difference is *inside* the idle-heartbeat line: on Windows it appends the console→bridge **reader-thread** stats (`reader[alive=… reads=… empties=…]`, from `NtHostTerminal.stdin_reader_stats`) — diagnostic for the Windows-only blocking-`ReadFile` reader; on macOS/Linux there is no bridge thread (the reactor `select`s the tty fd directly), so the same line shows `reader[n/a]`. Everything else is identical.

## Applying Code Changes to a Running aGiTrack

- **A code fix on disk does not take effect until the running aGiTrack process is restarted.** Python loads modules into memory at process start, so a long-lived proxy keeps executing the code it started with even after the source file changes. This is the usual reason a just-merged fix "still doesn't work": the file is fixed; the live process is stale. The remedy is to **fully quit and relaunch aGiTrack** — not reinstall.
- The dev's `agitrack` command is typically a **pipx editable install**: `~/.local/bin/agitrack` → pipx venv whose `__editable__.agitrack-*.pth` maps the `agitrack` package to the base-repo working tree (`/Users/s.wang9/Code/agitrack/agitrack`). So a fix that aGiTrack auto-merges into the checked-out base branch is immediately on disk for the installed command — no reinstall needed, just a restart. (Editable/source-linked vs package installs are also what the Self-Update section keys off.)
- Confirm the base repo's checked-out branch before assuming a fix is live: aGiTrack auto-merges into whatever branch the base repo is on (often `dev`), and `main`/release tags can lag well behind. If the base repo is checked out to a branch without the fix, the editable install loads unfixed code. The pipx-recorded version number can also be stale relative to `agitrack --version`.
